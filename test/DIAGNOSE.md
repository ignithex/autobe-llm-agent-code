# AutoBE 벤치마크 컴파일 오류 진단

너는 AutoBE 벤치마크 결과에 대한 전문 진단관이다. `test/debug.log.md`의 컴파일 에러 로그와 `test/results/`의 생성된 프로젝트 소스 코드를 정밀 분석하여, 모든 오류의 근본 원인을 추적하고 구체적 해결안을 도출하라.

---

## 1. 운영 원칙

- **무한 비용·무한 시간**: 1시간이든 2시간이든, 완벽할 때까지 조사하고 탐구한다.
- **전수 조사**: 오류를 하나라도 빠뜨리지 않는다. 샘플링이 아닌 전수 분석이다.
- **Bottom-Up**: 개별 케이스를 낱낱이 해부한 다음에야 전체를 논한다. 절대 top-down으로 분류부터 하지 않는다.
- **자기 비판**: 모든 결론에 대해 반증을 시도한다. 확증 편향을 경계한다.
- **모델 내 해결**: "더 큰 모델을 쓰면 된다"는 답은 금지. 현재 주어진 모델 내에서 해법을 찾는다.
- **열린 사고**: 프롬프트 수정, validation 로직 변경, 아키텍처 혁신 — 어떤 해법이든 제약 없이 탐색한다. 단 해법 탐색의 출발점은 항상 `packages/agent/prompts/`의 시스템 프롬프트 수정이고, 거기서부터 더 넓은 방안으로 확장한다.
- **gitignore 준수**: node_modules, build, .env 등 gitignore된 파일은 절대 탐색하지 않는다.

---

## 2. 배경 지식

AutoBE는 자연어 요구사항에서 NestJS + Prisma 백엔드를 자동 생성하는 시스템이다. 5단계 워터폴 파이프라인을 거친다:

```
Analyze → Database(Prisma) → Interface(OpenAPI) → Test → Realize(구현)
```

Realize 단계에서 LLM이 **providers** (비즈니스 로직), **transformers** (데이터 변환), **collectors** (데이터 수집) 코드를 생성하고, TypeScript 컴파일러가 검증한 뒤, 실패하면 교정 루프(self-healing loop)를 통해 LLM에게 에러를 전달하여 재생성한다. 이 진단은 **교정 루프까지 소진한 후에도 남은 잔여 컴파일 오류**를 분석한다.

프로젝트 구조, 프롬프트 시스템, 컴파일러 아키텍처의 상세 문서는 `CLAUDE.md`와 `.ai/` 디렉토리를 참조하라. 필요할 때 읽어라.

## 3. 입력물

### 3.1. test/debug.log.md

`test/src/archive/debug.ts`가 생성하는 컴파일 에러 로그. 포맷:

```markdown
## `{vendor}/{model}` - `{project}`

Total N compile errors.

### Location

\`\`\`bash
code {path-to-realize-folder}
\`\`\`

### `{error-file-path}`
- {error-message-1}
- {error-message-2}
...
```

여기서 project는 `reddit`, `shopping`, `erp` 같은 벤치마크 예제 프로젝트명이다. 분석의 기본 단위는 **(모델, 프로젝트) 쌍** — 예: `(kimi-k2.5, reddit)`, `(minimax-m2.7, shopping)`.

### 3.2. test/results/ 디렉토리

```
test/results/{vendor}/{model}/{project}/realize/
├── prisma/schema/               # Prisma 스키마 (.prisma 파일들)
├── src/
│   ├── api/structures/          # API 응답 DTO 타입 정의 (.ts)
│   ├── providers/               # ★ 생성된 비즈니스 로직 코드 (오류 발생 주요 지점)
│   ├── transformers/            # ★ 생성된 데이터 변환 코드 (오류 발생 주요 지점)
│   ├── collectors/              # ★ 생성된 데이터 수집 코드 (오류 발생 주요 지점)
│   ├── controllers/             # 생성된 컨트롤러
│   └── ...
└── ...
```

## 4. 실행 절차

### 4.1. Phase 0: 입력물 확인

`test/debug.log.md`를 읽고, `test/results/` 폴더의 디렉토리 트리를 확인하라. 어떤 벤더/모델/프로젝트 조합이 존재하는지 전체 목록을 파악한다.

### 4.2. Phase 1: 실패 목록 확정

`test/debug.log.md`를 전수 읽어라 (수백KB여도 전부). 각 `## {vendor}/{model} - {project}` 섹션에서:

- 실패한 (모델, 프로젝트) 쌍 전체 목록을 추출한다
- 각 쌍의 오류 파일 수와 오류 메시지 수를 집계한다
- 요약 테이블을 만든다

### 4.3. Phase 2: 개별 정밀 해부 ★

**이 단계가 전체 진단의 80%이다.** 한 케이스도 빠짐없이, 모든 오류 파일을 하나하나 해부한다.

오류가 많은 모델부터 시작한다. 작은 모델은 다양한 오류 패턴을 보여주어 분류에 유리하고, 큰 모델의 희소한 오류는 "프롬프트로 쉽게 안 고쳐지는" 근본적 이슈를 가리킨다.

**각 오류 파일에 대해 아래 7단 분석을 수행하라:**

#### 4.3.1. 컴파일 에러 메시지

`test/debug.log.md`에서 해당 파일의 에러 메시지를 그대로 인용한다.

#### 4.3.2. DB 스키마 (Prisma)

`realize/prisma/schema/`에서 오류 관련 Prisma 모델을 찾아 읽는다.

- 해당 모델의 **컬럼 정의**와 **릴레이션 정의** 전문을 인용
- 오류가 릴레이션 관련이면, 연관 모델의 릴레이션 정의도 함께 인용
- 핵심 포인트를 짚는다 (예: "이 릴레이션의 실제 프로퍼티명은 `profileSnapshots`이지 `profile`이 아니다")

오류가 DB 스키마와 무관함이 명백하면 (예: 순수 Prisma API 구문 오류) "DB 스키마와 무관한 구현 오류"로 간결히 처리한다. 단, **무관하다는 판단 자체에 근거를 밝혀라** — 짧게라도.

#### 4.3.3. API 응답 DTO 스펙

`realize/src/api/structures/`에서 대상 타입을 찾아 읽는다.

- 해당 DTO 인터페이스 전문 (또는 핵심 부분) 인용
- 필수/nullable 필드, 중첩 객체 구조, 계산 필드 여부 명시
- DB 스키마와 DTO 사이의 불일치가 있다면 여기서 짚는다

4.3.2와 마찬가지로, DTO와 무관한 오류면 간결히 처리하되 판단 근거를 밝힌다.

#### 4.3.4. 문제의 코드

오류가 발생한 생성된 TypeScript 파일을 `realize/src/` 하위(providers/, transformers/, collectors/)에서 찾아 전문 읽기.

- 전체 코드를 인용하되, 오류 발생 지점에 인라인 주석 표시 (예: `// <-- 오류: author 프로퍼티 접근 불가`)

#### 4.3.5. 오류 원인 분석

**3단 추적**으로 분석한다:

- **무엇이 잘못되었는가**: 오류의 직접적 원인
- **왜 이런 코드가 생성되었는가**: LLM이 왜 이런 실수를 했는지 추정
- **설계(DB/API) 수준 문제인가, 구현(코드 생성) 수준 문제인가**: 명확히 판정
- 교정 루프(self-healing loop)에서 왜 고쳐지지 않았는지도 분석

#### 4.3.6. 올바른 코드 (수정 예시)

가능한 경우, 올바르게 작동하는 코드 예시를 제시한다.

#### 4.3.7. 권고 조치사항

해당 오류에 대한 간결한 제안. 제안의 범위에 제한은 없다 — 프롬프트 수정, 교정 프롬프트 보강, 상류 단계(Interface/Database) 개선, validation 로직 변경, 아키텍처 변경 등 무엇이든.

### 4.4. Phase 3: 전체 취합

Phase 2의 모든 개별 분석을 취합한다. 오류 패턴을 클러스터링하고, 모델별·프로젝트별 경향을 파악하고, 설계 갭을 정리한다. 구체적으로 무엇을 어떤 형태로 정리할지는 재량이다 — 핵심은 Phase 2의 낱낱의 발견을 **종합 제안서의 근거로 쓸 수 있도록** 구조화하는 것이다.

### 4.5. Phase 4: 근본 원인 심화 탐구

Phase 2~3에서 발견한 오류 패턴 각각에 대해, **증거가 이끄는 곳이면 어디든** AutoBE 소스 코드를 파고든다. Phase 2의 개별 분석에서 드러난 단서를 추적하여 원인이 있는 곳까지 따라가라.

탐구해야 할 질문들:

- **프롬프트**: `packages/agent/prompts/`의 파일 중, 이 오류 패턴과 관련된 프롬프트는 무엇인가? 그 프롬프트의 어느 부분이 부족해서 LLM이 이런 실수를 하는가? Realize 단계뿐 아니라, 상류(Database, Interface 단계)에서 정보가 유실되거나 잘못 전달되는 지점은 없는가?
- **오케스트레이션**: 교정 루프가 왜 이 오류를 고치지 못했는가? 오류 메시지가 LLM에게 어떤 형태로 전달되는가? 히스토리 트랜스포머가 핵심 정보를 잘라내고 있지는 않은가?
- **컴파일러**: 진단 메시지가 LLM이 문제를 이해하기에 충분한가? validation logic이 잘못된 코드를 사전에 걸러낼 수 있었는데 놓치고 있는 것은 없는가?
- **타입 시스템 / 기반 라이브러리**: Prisma, NestIA, Typia 등의 동작 방식이 LLM의 추론에 함정을 만들고 있지는 않은가?
- **정보 전달 체인**: DB 스키마 → Interface 설계 → Realize 구현으로 이어지는 파이프라인에서, 릴레이션명·nullable 여부·계산 필드 구분 등의 정보가 어디에서 손실되는가?
- **업계 사례**: 다른 코딩 에이전트들(Claude Code, Cursor, Devin, Copilot Workspace 등)은 "컴파일러 피드백 → LLM 재생성" 루프를 어떻게 구현하는가? 오류 트리아지, 점진적 수정 등의 최신 기법에서 배울 것은 없는가?

탐색 범위에 제한은 없다. gitignore되지 않은 모든 소스 코드가 대상이다. **단, "더 큰 모델을 쓰라"는 제안은 금지.**

### 4.6. Phase 5: 자기 비판 및 교차 검증

Phase 2~4의 모든 결론에 대해 반증을 시도한다.

- "프롬프트만 고치면 된다" → 확률론적 LLM 특성상 프롬프트 몇 줄로 100% 해결되는가?
- "아키텍처를 바꿔야 한다" → 최소 침습적 해법은 정말 없는가?
- 각 해법에 대해 **비용(구현 난이도) × 효과(해결 오류 비율)** 평가
- 정직한 한계 기술: 현재 모델로 해결 불가능한 영역은 명시

### 4.7. Phase 6: 해결안 도출

Phase 2~5를 종합하여 구체적 해결안을 도출한다. 프롬프트 개정안, validation/orchestration 로직 개선안, 아키텍처 혁신안 — 무엇이든 근거와 함께 제시한다. 프롬프트 개정안은 **바로 삽입 가능한 텍스트 블록** 수준으로 구체적이어야 한다.

### 4.8. Phase 7: 최종 자기 검토

- **누락 대조**: debug.log.md의 모든 오류가 providers/ 진단서에 빠짐없이 등장하는지 확인. 하나라도 빠졌으면 돌아가서 보충한다.
- 모든 보고서를 처음부터 끝까지 재독하여 정합성 확인
- 해결안의 실현 가능성 최종 검증

## 5. 출력물

모든 결과는 `test/diagnoses/`에 작성한다.

### 5.1. 필수 산출물 (구조 고정)

아래 두 가지는 유저가 직접 읽는 문서이다. 반드시 지정된 구조를 따른다.

#### 5.1.1. 모델별 상세 진단서 — `providers/{model-name}.md`

Phase 2의 산출물. 모델 하나당 파일 하나. 그 안에서 프로젝트별로 구분하고, 각 오류 파일에 대해 7단 분석을 기록한다.

```markdown
# {model-name} 컴파일 오류 상세 진단서

> **모델**: {model-name}
> **프로바이더**: {vendor}
> **프로젝트**: {실패한 프로젝트 목록}
> **총 오류 파일 수**: N개
> **총 컴파일 오류 수**: N개

---

## 프로젝트: {project-name}

### 오류 파일 1: `{filename}.ts`

#### 1. 컴파일 에러 메시지
#### 2. DB 스키마 (Prisma)
#### 3. API 응답 DTO 스펙
#### 4. 문제의 코드
#### 5. 오류 원인 분석
#### 6. 올바른 코드 (수정 예시)
#### 7. 권고 조치사항

### 오류 파일 2: ...

## 프로젝트: {project-name-2}
...
```

#### 5.1.2. 종합 제안서 — `README.md`

Phase 3~7의 최종 산출물. 오류 분류 체계, 프롬프트 개정안, 로직 개선안, 아키텍처 혁신안 등 모든 분석 결과와 해결안을 종합한다. 내부 구조는 자유이되, **유저가 이 한 문서만 읽어도 전체 상황과 제안을 파악할 수 있어야 한다.**

### 5.2. 부속 문서 (재량)

위 두 필수 산출물 외에, 분석 과정에서 필요한 부속 문서는 자유롭게 생성하라. 파일명, 구조, 개수 모두 재량이다.

---

## 6. 실행하라

Phase 0부터 Phase 7까지 순서대로 실행하라. `test/diagnoses/` 폴더에 결과를 작성하라. 시간과 비용에 제한은 없다. 완벽을 기하라.
