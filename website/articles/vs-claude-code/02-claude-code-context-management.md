# Claude Code 심층 분석: 컨텍스트 관리, 캐싱, 프롬프트 엔지니어링

> Claude Code의 컨텍스트 관리 시스템에 대한 기술 리뷰이며, 유출된 소스 코드(512,000줄)에서 도출하였다. 이 보고서는 5계층 컨텍스트 압축 파이프라인, 시스템 프롬프트 아키텍처, 토큰 관리 전략, 프롬프트 캐싱 최적화를 다룬다. 모든 평가는 관찰 가능한 코드 패턴에 근거한다.

---

## 1. 5계층 컨텍스트 압축 파이프라인

Claude Code는 200,000 토큰 컨텍스트 윈도우 내에서 동작한다. 광범위한 tool 사용(파일 읽기, 검색 결과, bash 출력)을 동반하는 길고 다단계적인 코딩 세션을 장려하는 tool에서, 컨텍스트 소진은 이론적 우려가 아니라 일상적인 운영 과제이다. Claude Code는 대화가 길어짐에 따라 컨텍스트 크기를 점진적으로 줄이는 5계층 압축 파이프라인으로 이를 해결한다.

이 파이프라인은 흔히 4계층으로 설명된다. 유출된 소스는 다른 네 계층과 다른 세분성에서 작동하는 다섯 번째 계층(계층 0: Tool 결과 예산)을 드러내어, 5계층 시스템을 구성한다.

### 1.1 계층 0: Tool 결과 예산

**작동 원리**: 메시지별 합산 시행과 동결된 교체.

대화 수준의 압축이 발생하기 전에, 개별 tool 결과가 생성 시점에서 제한된다. 이것이 계층 0인 이유는 여러 메시지에 *걸쳐서*가 아니라 단일 메시지 *내에서* 작동하기 때문이다.

메커니즘의 중심에는 `ContentReplacementState`가 있는데, 이는 다음을 추적하는 데이터 구조이다:

- 현재 어시스턴트 턴 내 tool 결과의 누적 크기
- 어떤 tool 결과가 얼마나 잘렸는지
- 어떤 잘라내기가 "동결"(영구적으로 확정)되었는지

Tool 결과가 생성되면, 시스템은 세 가지 임계값(동반 보고서의 3계층 예산: tool당 50KB, 합산 200KB, 절대 100K 토큰)에 대해 검사한다. 결과가 어떤 임계값이든 초과하면 잘리고 잘라내기가 `ContentReplacementState`에 기록된다.

핵심적인 설계 결정은 **동결된 교체**이다: tool 결과가 한번 잘리면 잘라내기는 영구적이다. 이후 상위 계층에서의 컨텍스트 압축이 토큰 예산을 확보하더라도, 잘린 tool 결과는 복원되지 않는다. 이는 진동 시나리오를 방지한다:

1. Tool 결과가 계층 0에서 잘린다 (토큰 절약)
2. 나중에 계층 4(autocompact)가 실행되어 토큰을 확보한다
3. 동결된 교체가 없으면, 시스템이 tool 결과의 잘라내기를 "해제"하여 확보된 토큰을 소모할 수 있다
4. 이것이 다시 계층 4를 트리거하여 순환을 만들 수 있다

동결된 교체는 약간의 정보 손실을 감수하고 이 순환을 끊는다. 이 트레이드오프는 합당하다: 잘린 tool 결과는 일반적으로 큰 출력(로그 파일, 많은 매치를 가진 검색 결과)이었으며, 잘린 내용이 모델의 추론에 핵심적일 가능성은 낮았다.

### 1.2 계층 1: Snip (HISTORY_SNIP)

**작동 원리**: 특정 메시지 내용의 대상 지정 제거.

계층 1은 파이프라인에서 가장 수수께끼 같은 계층이다. 유출된 소스에는 상수 `HISTORY_SNIP`과 반환 타입 시그니처 `{messages, tokensFreed, boundaryMessage}`를 포함한 참조가 있지만, 구현 자체는 부재하다. 가장 가능성 있는 설명은 번들러에 의한 사용하지 않는 코드 제거(DCE)이다: 번들 시점에 비활성화된 feature flag 뒤에 구현이 게이트되어 있어, 번들러가 코드를 완전히 제거한 것이다.

남은 참조에서 추론할 수 있는 것:

- **Snip은 개별 메시지에 대해 작동하며**, 메시지 내의 특정 콘텐츠 블록을 제거하거나 교체한다. 이는 계층 2(tool 결과에 대해 작동)와 계층 3(전체 대화 세그먼트에 대해 작동)과 구별된다.
- **`boundaryMessage` 반환 값**은 Snip이 대화 히스토리에서 경계를 식별함을 시사한다. 경계 이전의 메시지가 snipping 대상이고, 이후의 메시지는 보존된다. 이는 Snip이 최신성 편향 압축임을 의미한다: 오래된 메시지가 먼저 snip된다.
- **`tokensFreed` 추적**은 Snip이 얼마나 많은 토큰을 확보했는지를 보고하여, 파이프라인이 상위 계층에서의 추가 압축이 필요한지를 결정할 수 있게 함을 나타낸다.

유출된 소스에서 이 계층이 부재한다는 것은 다음 중 하나를 의미한다:
- 제거되었지만 참조가 완전히 정리되지 않은 사용 중단 기능
- 아직 활성화되지 않은 개발 중인 기능
- 특정 배포 구성(예: 기업용)에서만 사용 가능한 기능

어느 쪽이든, 부재는 프로덕션 파이프라인이 사실상 4계층 시스템(0, 2, 3, 4)으로 작동하며, 계층 1은 플레이스홀더임을 의미한다.

### 1.3 계층 2: Microcompact

**작동 원리**: 경량 대상 지정 압축을 위한 세 가지 하위 경로.

Microcompact는 여러 메시지에 걸쳐 작동하는 첫 번째 계층이다. 오래된 메시지의 tool 결과를 대상으로 하여, 더 짧은 요약으로 교체하거나 완전히 제거한다. 이 계층은 컨텍스트 조건에 따라 선택되는 세 가지 하위 경로를 가진다:

#### 하위 경로 A: 시간 기반 Microcompact (Cold Cache)

프롬프트 캐시가 "콜드"(이전 요청에서 캐시된 컨텍스트 없음)이거나 메시지 간 60분 이상의 간격이 있을 때 트리거된다. 이 시나리오에서 오래된 tool 결과가 로컬에서 삭제된다. 내용이 `[Tool result cleared - see more recent context]` 같은 플레이스홀더로 대체된다.

60분 간격 휴리스틱은 한 시간 이상 전의 tool 결과가 현재 작업과 거의 관련이 없다는 관찰에 기반한다. 사용자가 다른 문제나 파일로 이동했을 가능성이 높고, 오래된 tool 결과는 가치를 제공하지 않으면서 토큰을 소모하고 있다.

#### 하위 경로 B: 캐시된 Microcompact (cache_edits)

기술적으로 가장 참신한 하위 경로이다. 프롬프트 캐시가 웜(이전 요청이 서버에 캐시된 컨텍스트를 설정)일 때, 이 경로는 **캐시 무효화 없이 서버 측 삭제**를 수행한다.

메커니즘은 다음과 같이 작동한다:

1. 클라이언트가 제거해야 할 tool 결과를 식별한다
2. 메시지 히스토리에서 제거하는 대신(이는 토큰 시퀀스를 변경하고 캐시를 무효화할 것이다), 클라이언트가 API에 `cache_edits` 명령을 보낸다
3. 서버가 KV 캐시를 재계산하지 않고 캐시된 컨텍스트에서 지정된 내용을 논리적으로 삭제한다
4. 후속 요청은 수정된 캐시를 사용하여 토큰과 캐시 계산 비용 모두를 절약한다

이것은 놀라운 최적화이다. 일반적인 캐싱 시스템에서 캐시된 내용의 수정은 캐시 무효화와 재계산을 요구한다. `cache_edits` 메커니즘은 이 비용 없이 캐시에서 내용을 수술적으로 제거할 수 있게 한다. 트레이드오프는 서버가 이 비표준 캐싱 동작을 지원해야 하므로, Claude Code가 Anthropic API의 캐싱 구현에 밀접하게 결합된다는 것이다.

#### 하위 경로 C: API 기반 컨텍스트 관리 (Anthropic 전용)

세 번째 하위 경로는 컨텍스트 관리를 전적으로 서버에 위임한다. 클라이언트가 전체 대화 히스토리와 컨텍스트 관리가 필요하다는 힌트를 보내면, 서버가 자체 압축 전략을 적용한다. 이 경로는 "ant-only" 플래그 뒤에 게이트되어 있어, 내부 Anthropic 배포 또는 특정 API 티어에서만 사용 가능함을 시사한다.

이 하위 경로는 클라이언트 관점에서 가장 단순하지만 가장 투명하지 않다. 클라이언트는 서버가 무엇을 제거하거나 요약할지 모르며, 컨텍스트 관련 문제의 디버깅이 더 어려워진다.

### 1.4 계층 3: Context Collapse (CONTEXT_COLLAPSE)

**작동 원리**: 전체 히스토리에 대한 읽기 시점 투영(projection), 변이가 아님.

Context Collapse는 아키텍처적으로 가장 흥미로운 계층이다. 대화가 컨텍스트 예산의 90%를 넘어 성장하면, Context Collapse가 활성화되어 전체 히스토리보다 상당히 짧은 대화 히스토리의 *투영된 뷰*를 생성한다.

핵심적인 설계 통찰은 Context Collapse가 변이가 아닌 **읽기 시점 투영**이라는 것이다. 전체 대화 히스토리는 메모리에 보존된다. 시스템이 API에 요청을 보내야 할 때, 전체 히스토리를 collapse된 형태로 투영한다. 이는 다음을 의미한다:

- 전체 히스토리를 언제든 복원할 수 있다 (예: 사용자가 대화 내보내기를 요청하는 경우)
- 서로 다른 목적에 대해 서로 다른 투영을 계산할 수 있다 (예: 메인 쿼리에 대해서는 더 공격적인 collapse, coordinator 하위 agent에 대해서는 덜 공격적인 collapse)
- 투영은 멱등(idempotent)이다: 두 번 적용해도 한 번 적용한 것과 동일한 결과를 낸다

투영 알고리즘은 다음과 같이 작동한다:

1. **Collapse 경계 식별**: 경계 이전의 메시지가 collapse 대상이다. 경계는 최근 컨텍스트를 보존하면서 collapse할 수 있는 히스토리의 양을 최대화하도록 계산된다.
2. **Collapse된 메시지 요약**: 경계 이전의 메시지가 커밋 로그 스타일의 요약으로 대체된다. 이 요약은 핵심 결정, 파일 수정, 오류 해결을 보존하면서 장황한 tool 출력과 중간 추론을 폐기한다.
3. **요약을 "커밋 로그"로 보존**: 요약은 후속 collapse에서도 살아남는 영속 아티팩트로 저장된다. Context Collapse가 나중에 다시 실행되면, 이전 collapse의 요약은 보존되고 새 요약은 마지막 collapse 이후의 메시지만 커버한다.

**단계적 임계값**:
- **예산의 90%**: Context Collapse가 "단계적(staged)" 모드로 활성화된다. 투영이 계산되지만 아직 사용되지 않는다. 시스템은 전체 히스토리로 계속 동작하지만 임박한 collapse를 준비한다.
- **예산의 95%**: Context Collapse가 "차단(blocking)" 모드로 활성화된다. 투영이 API 요청에 대해 전체 히스토리를 대체한다. 시스템은 더 이상 압축 없이 동작할 수 없다.

**Autocompact와의 상호작용**: Context Collapse가 활성화되면 autocompact(계층 4)가 비활성화된다. 이는 두 계층이 동시에 같은 히스토리를 압축하려 시도하여 비일관적인 결과를 생성할 수 있는 상황을 방지한다. Context Collapse가 우선권을 가지는데, 더 예측 가능하기 때문이다(결정적 투영 vs. LLM 기반 요약).

### 1.5 계층 4: Autocompact

**작동 원리**: 구성 가능한 토큰 임계값에서의 LLM 기반 요약.

Autocompact는 가장 공격적인 압축 계층이다. 컨텍스트가 약 167,000 토큰(200K 윈도우 - 응답용 20K 예약 - 13K 버퍼)에 도달하면, 시스템이 LLM을 호출하여 대화 히스토리를 요약한다.

요약 프로세스:

1. **요약 대상 콘텐츠 선택**: 오래된 메시지, tool 결과, 중간 추론이 대상으로 식별된다. 최근 메시지(구성 가능한 최신성 윈도우 내)는 그대로 보존된다.
2. **요약 생성**: LLM이 선택된 내용을 요약하도록 프롬프트되며, 핵심 결정, 파일 변경, 오류 해결을 보존한다. 프롬프트는 대화적 맥락보다 실행 가능한 정보(파일 경로, 함수명, 오류 메시지)의 보존을 강조한다.
3. **콘텐츠를 요약으로 교체**: 선택된 내용이 LLM이 생성한 요약으로 대체된다. 이는 변이(Context Collapse의 투영과 달리)이므로 원본 내용이 유실된다.

**세션 메모리 압축**: Autocompact는 "세션 메모리" -- 컨텍스트 압축 간에 지속되는 대화의 실행 요약 -- 도 압축한다. Autocompact가 실행되면, 요약된 내용의 정보를 반영하여 세션 메모리가 업데이트되며, 여러 차례의 압축에서도 중요한 컨텍스트가 살아남도록 보장한다.

**서킷 브레이커**: 3회 연속 autocompact 실패(예: LLM이 일관성 있는 요약을 생성하지 못하거나 요약이 원본보다 큰 경우) 후, 시스템은 세션의 나머지 기간 동안 autocompact를 비활성화하고 Context Collapse로 폴백한다. 이는 실패하는 요약 모델이 압축 시도의 무한 루프를 유발하는 시나리오를 방지한다.

### 1.6 계층 캐스케이드: 계층 간 상호작용

계층들은 순서대로(0부터 4까지) 평가되며, 이전 계층이 충분한 토큰을 확보하지 못한 경우에만 다음 계층이 활성화된다:

```
계층 0: Tool 결과 예산
  |
  v (계층 0 후에도 예산 초과인 경우)
계층 1: Snip (현재 비활성)
  |
  v (계층 1 후에도 예산 초과인 경우)
계층 2: Microcompact
  |
  v (계층 2 후에도 예산 초과인 경우)
계층 3: Context Collapse
  |
  v (계층 3 후에도 예산 초과이고, collapse가 차단 모드가 아닌 경우)
계층 4: Autocompact
```

실제로 캐스케이드는 엄밀히 순차적이지 않다. 계층 0과 2는 독립적으로 활성화될 수 있고(계층 0은 모든 tool 결과에서, 계층 2는 특정 트리거에서), 계층 3과 4는 상호 배타적이다(collapse가 autocompact를 비활성화함).

#### 90%/93%/95% 경쟁 조건

Context Collapse(계층 3)와 Autocompact(계층 4) 사이에 임계값 경계 근처에서 미묘한 상호작용이 존재한다:

- 예산의 83.5% (167K/200K)에서: Autocompact 활성화
- 예산의 90%에서: Context Collapse가 단계적 모드로 활성화
- 예산의 93%에서: autocompact(아직 실행 중)와 collapse(새로 단계적) 간 잠재적 경쟁
- 예산의 95%에서: Context Collapse가 차단 모드로 활성화되어 autocompact를 비활성화

경쟁 조건은 93%에서 발생한다: 컨텍스트가 90%를 넘을 때 autocompact가 실행 중이면, 두 계층이 동시에 압축을 시도할 수 있다. 해결 방법은 Context Collapse의 단계적 phase가 투영을 *준비*만 하고 적용하지는 않는다는 것이다. 투영은 95%(차단 모드)에서만 적용되며, 이 시점에서 autocompact가 강제 비활성화된다. 이것은 두 계층이 모두 활성 상태이지만 컨텍스트의 서로 다른 측면에서 작동하는 좁은 윈도우(90-95%)를 남긴다.

코드베이스는 이를 올바르게 처리하지만 미묘하다. 정확성은 단계적 collapse가 메시지 히스토리를 수정하지 않는다는 불변량에 의존하며, 이는 읽기 시점 투영 설계에 의해 유지된다. 이 불변량이 위반되면(예: 투영을 조기에 적용하는 미래의 최적화에 의해) 경쟁 조건이 데이터 손실을 유발할 수 있다.

## 2. 시스템 프롬프트 아키텍처

### 2.1 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 센티널

Claude Code의 시스템 프롬프트는 센티널 문자열에 의해 두 영역으로 나뉜다:

```
[정적 시스템 프롬프트 내용]
SYSTEM_PROMPT_DYNAMIC_BOUNDARY
[동적 시스템 프롬프트 내용]
```

**정적 내용** (센티널 위):
- 정체성 및 역량 선언
- 핵심 행동 지침
- 보안 정책 및 제약
- Tool 사용 가이드라인
- 출력 형식 규칙

**동적 내용** (센티널 아래):
- 세션별 가이드
- 환경 정보 (OS, 셸, 작업 디렉토리)
- 사용자 선호 (언어, 출력 스타일)
- MCP 서버 지침
- 메모리 (이전 세션의 지속적 사용자 컨텍스트)

센티널은 캐싱에서 핵심적인 역할을 한다. 프롬프트 캐싱은 프롬프트의 접두사를 해싱하고 일치하는 접두사에 대해 캐시된 결과를 제공하는 방식으로 작동한다. 모든 세션 가변 내용을 센티널 *이후에* 배치함으로써, 정적 접두사가 세션 간에(그리고 같은 조직 내 사용자 간에도) 동일하게 유지된다. 이는 캐시 적중률을 최대화한다:

- **사용자 간 캐싱**: 같은 Claude Code 버전의 모든 사용자가 동일한 정적 접두사를 공유하여 전역 캐시 재사용이 가능하다.
- **세션 간 캐싱**: 같은 사용자의 서로 다른 세션이 정적 접두사를 공유하여 세션 간 캐시 재사용이 가능하다.
- **세션 내 안정성**: 동적 내용은 세션 내에서 드물게 변경되므로(주로 작업 디렉토리 변경이나 MCP 서버 설치 시), 전체 프롬프트도 세션 내 요청 간에 캐시 가능한 경우가 많다.

### 2.2 캐시 범위

시스템 프롬프트 섹션에는 얼마나 광범위하게 캐시될 수 있는지를 제어하는 캐시 범위가 주석으로 달려 있다:

- **`global`**: 모든 사용자와 조직에 걸쳐 캐시됨. 진정으로 보편적인 내용(핵심 지침, 보안 정책)만 이 범위를 받는다.
- **`org`**: 조직 내에서 캐시됨. 조직별 정책과 구성이 이 범위를 받는다.
- **`session`**: 단일 세션 내에서 캐시됨. 세션별 컨텍스트(환경 정보, 메모리)가 이 범위를 받는다.
- **`null`**: 캐시되지 않음. 요청 간 변경되는 내용(예: 대화 상태에 의존하는 동적 컨텍스트)이 이 범위를 받는다.

범위 시스템은 계층적 캐싱 전략을 가능하게 한다:

```
전역 캐시 (모든 사용자가 공유, ~1시간 TTL)
  └── 조직 캐시 (조직 사용자가 공유)
       └── 세션 캐시 (세션별)
            └── 요청별 내용 (캐시되지 않음)
```

이 계층 구조는 LLM 추론의 경제학에 잘 설계되어 있다. 처리 비용이 가장 높은 내용(긴 정적 지침)이 가장 넓은 범위에서 캐시되고, 가장 가변적인 내용(요청별)은 작게 유지된다.

### 2.3 섹션 레지스트리와 DANGEROUS_uncachedSystemPromptSection

시스템 프롬프트 내용은 명명된 섹션의 레지스트리로 구성된다. 각 섹션은 다음을 가진다:

- 이름 (디버깅 및 로깅용)
- 캐시 범위
- 내용 (문자열 또는 문자열을 생성하는 함수)
- 선택적 조건 (섹션이 포함될지 결정하는 함수)

대부분의 섹션은 요청 간 캐싱을 가능하게 하는 캐시 범위로 등록된다. 그러나 한 가지 특별한 등록 함수가 존재한다: `DANGEROUS_uncachedSystemPromptSection`. "DANGEROUS" 접두사는 강제 함수(보안 아키텍처의 policy-as-type 패턴과 유사)이다: 캐시되지 않는 내용을 시스템 프롬프트에 추가하면 캐싱 성능이 저하된다는 것을 개발자가 인지하도록 강제한다.

유출된 소스에서 MCP(Model Context Protocol) 서버 지침만이 이 등록을 사용한다. 이는 합리적이다: MCP 서버 구성은 요청 간에 변경될 수 있고(사용자가 세션 중간에 새 MCP 서버를 연결하는 경우), 오래된 MCP 구성을 캐시하면 tool 디스커버리 실패가 발생할 수 있다. 하지만 비용은 상당하다 -- 캐시되지 않는 섹션은 캐시 미스 시 API가 시스템 프롬프트를 처음부터 재처리하도록 강제한다.

### 2.4 동적 주입 포인트

시스템 프롬프트의 동적 영역은 여러 주입 포인트를 지원한다:

- **`session_guidance`**: 오래 지속되는 세션 컨텍스트. 예: "사용자가 React 프로젝트에서 작업 중" 또는 "JavaScript보다 TypeScript 선호."
- **`memory`**: 지속적 세션 간 메모리. 사용자 선호, 프로젝트 관례, 자주 사용하는 파일 경로를 포함한다. `~/.claude/memory.md` 또는 유사한 곳에서 로드된다.
- **`env_info`**: 환경 정보: 운영 체제, 셸, 작업 디렉토리, git 상태, node/python 버전.
- **`language`**: 응답에 대한 사용자의 선호 언어.
- **`output_style`**: 형식 선호 (상세, 간결, 마크다운, 일반 텍스트).
- **`mcp_instructions`**: 연결된 MCP 서버에 대한 구성 및 tool 설명.
- **`project_instructions`**: `CLAUDE.md` 파일의 프로젝트별 지침.
- **`custom_instructions`**: 설정에서 사용자 정의 커스텀 지침.

### 2.5 사용자 컨텍스트 주입: 첫 번째 사용자 메시지의 system-reminder

주목할 만한 아키텍처 결정은 사용자 제공 컨텍스트(`CLAUDE.md` 파일 내용 등)가 시스템 프롬프트에 주입되지 않는다는 것이다. 대신 첫 번째 사용자 메시지에서 `<system-reminder>` 태그로 주입된다.

이것은 여러 함의를 가진 의도적인 선택이다:

1. **캐시 보존**: 사용자별 내용을 시스템 프롬프트에 추가하면 캐시가 파편화될 것이다. 사용자 메시지에 배치함으로써 시스템 프롬프트가 전역적으로 캐시 가능한 상태를 유지한다.
2. **우선순위 정렬**: 사용자 메시지의 내용은 시스템 프롬프트의 내용보다 낮은 우선순위를 가진다(모델은 시스템 프롬프트 지침을 더 높은 권위로 취급함). 사용자 제공 내용을 사용자 메시지에 배치함으로써, 자체 보안 정책과 행동 제약(시스템 프롬프트 내)이 사용자가 `CLAUDE.md`에 넣은 것보다 우선하도록 보장한다.
3. **가시성**: `<system-reminder>` 태그는 대화 히스토리에서 보이므로 디버깅에 도움이 된다. 모델이 예상치 않게 동작하면 개발자가 대화를 검사하여 정확히 어떤 사용자 컨텍스트가 주입되었는지 확인할 수 있다.

### 2.6 전체 시스템 프롬프트 구성

유출된 소스에 기반한 전체 시스템 프롬프트 구성:

| 영역 | 내용 | 대략적 크기 |
|------|------|------------|
| 정적 | 정체성, 역량, 핵심 지침 | ~3-4K 문자 |
| 정적 | 보안 정책, 제약 | ~2-3K 문자 |
| 정적 | Tool 사용 가이드라인 | ~2-3K 문자 |
| 동적 | 환경 정보 | ~500-1K 문자 |
| 동적 | 세션 가이드 | ~500-2K 문자 |
| 동적 | 메모리 | ~500-2K 문자 |
| 동적 | 언어/출력 스타일 | ~200-500 문자 |
| 동적 | MCP 지침 | 0-5K+ 문자 |
| 동적 | 프로젝트 지침 | 0-5K+ 문자 |

합계: 정적 약 8-10K 문자 + 동적 2-10K+ 문자(구성에 따라 다름). 정적 부분은 안정적이고 캐시 가능하며, 동적 부분은 세션과 프로젝트에 따라 달라진다.

## 3. 토큰 관리

### 3.1 3계층 토큰 카운팅 전략

정확한 토큰 카운팅은 컨텍스트 관리(언제 압축할지 알기)와 비용 추정(사용자에게 토큰 사용량 보고)에 필수적이다. Claude Code는 3계층 전략을 사용한다:

**1계층: API 카운팅 (정확)**

각 API 응답 후, 서버가 요청에 대한 정확한 토큰 수(입력 토큰, 출력 토큰, 캐시 읽기 토큰, 캐시 쓰기 토큰)를 반환한다. 이 수치는 저장되어 가장 최근 요청에 대한 정답(ground truth)으로 사용된다.

**2계층: 대략적 추정 (4바이트당 1토큰)**

API 호출 사이(예: tool 결과 추가 후 다음 API 요청 전)에, 시스템은 간단한 휴리스틱을 사용하여 토큰 수를 추정한다: 4바이트당 1토큰. 이는 직렬화된 메시지 내용의 바이트 길이에 적용된다.

4바이트당 1토큰 비율은 주석에서 부정확한 것으로 인정되고 있다. 영어 산문의 경우 실제 비율은 3.5-4.5바이트당 1토큰에 가깝다(어휘에 따라 다름). 코드의 경우 비율이 상당히 달라질 수 있다:

- 공백이 많은 코드 (Python, YAML): 3바이트당 1토큰에 가까울 수 있음
- 기호가 많은 코드 (정규식, Perl): 5+ 바이트당 1토큰에 가까울 수 있음
- 비ASCII 내용 (유니코드 식별자, CJK 주석): 6+ 바이트당 1토큰에 가까울 수 있음

부정확함에도 불구하고, 4바이트당 1토큰 추정이 사용되는 이유는 빠르고(API 호출 불필요) 대부분의 경우 컨텍스트 오버플로를 피하기에 충분히 보수적이기 때문이다.

**3계층: 하이브리드 (마지막 API 사용량을 기준선으로)**

하이브리드 접근 방식은 가장 최근 API 호출의 정확한 토큰 수를 기준선으로 사용하고, 새 내용에서의 증분은 4바이트당 1토큰 휴리스틱으로 추정한다. 이는 순수 추정보다 더 나은 정확도를 제공한다. 기준선이 정확하고 증분만 추정되기 때문이다.

```
추정_합계 = 마지막_api_토큰_수 + (새_내용_바이트 / 4)
```

이 하이브리드 접근 방식은 컨텍스트 압축 임계값 검사에 사용된다. 정확한 트리거 시점이 수천 토큰 정도 달라질 수 있더라도, 대략 적절한 시점에 압축을 트리거하기에 충분히 정확하다.

### 3.2 200K 컨텍스트 윈도우의 예산 할당

200,000 토큰 컨텍스트 윈도우는 여러 영역으로 분할된다:

| 영역 | 토큰 | 목적 |
|------|------|------|
| 유효 예산 | 180,000 | 대화 + 시스템 프롬프트에 사용 가능 |
| 응답 예약 | 20,000 | 모델의 응답을 위해 예약됨 |
| Autocompact 임계값 | 167,000 | 초과 시 autocompact 트리거 |
| 경고 임계값 | 160,000 | 사용자에게 보이는 경고 트리거 |
| 차단 임계값 | 177,000 | 차단 context collapse 트리거 |
| 버퍼 | 13,000 | autocompact와 유효 예산 간 간격 |

20,000 토큰 응답 예약은 업계 표준으로는 넉넉하다(많은 시스템이 4,096이나 8,192를 사용함). 더 큰 예약은 Claude Code의 유스케이스를 반영한다: 모델이 코드 블록, 설명, 여러 tool 호출을 포함하는 긴 응답을 자주 생성하기 때문이다. 더 작은 예약은 빈번한 `max_tokens` 잘림을 유발하여 agent loop의 복구 메커니즘을 트리거하고 사용자 경험을 저하시킬 것이다.

Autocompact 임계값(167K)과 유효 예산(180K) 사이의 13,000 토큰 버퍼는 안전 마진을 제공한다. Autocompact는 실행에 시간이 걸리고(LLM 호출이 필요함), 그 시간 동안 대화가 계속 성장할 수 있다. 버퍼는 autocompact 실행 중 컨텍스트가 오버플로하지 않도록 보장한다.

### 3.3 Microcompact 패딩: 4/3 보수적 계수

Microcompact(계층 2)가 실행될 때, 필요한 감소량의 4/3(133%)에 해당하는 예산을 목표로 한다. 예를 들어, 10,000 토큰을 확보해야 하면 microcompact는 13,333 토큰을 목표로 한다.

4/3 계수는 추정 부정확성을 감안한다. 시스템이 개별 tool 결과에 대한 정확한 토큰 수를 가지고 있지 않으므로(대략적 추정만), 보수적 목표는 추정이 최대 25% 빗나가더라도 microcompact가 충분한 토큰을 확보하도록 보장한다.

이 패딩은 토큰 수가 추정인 개별 tool 결과에 대해 작동하는 microcompact에 특화되어 있다. 전체 대화에 대해 작동하는 autocompact는 가장 최근 API 호출의 정확한 토큰 수를 사용하므로 패딩이 필요하지 않다.

## 4. 프롬프트 캐싱 전략

프롬프트 캐싱은 Claude Code의 가장 중요한 성능 최적화 중 하나이다. Anthropic API는 프롬프트 접두사의 캐싱을 지원한다: 두 요청이 같은 접두사를 공유하면, 두 번째 요청이 첫 번째 요청의 캐시된 KV 계산을 재사용하여 지연 시간과 비용을 대폭 줄일 수 있다.

### 4.1 캐시 중단점 배치

Claude Code는 프롬프트의 전략적 위치에 캐시 중단점을 배치한다:

1. **정적 시스템 프롬프트 끝**: 핵심 지침의 사용자 간 전역 캐싱을 가능하게 한다.
2. **Tool 스키마 끝**: Tool 스키마는 세션 안정적(한 번 계산되고 고정됨)이므로 세션 내에서 캐시 가능하다.
3. **동적 시스템 프롬프트 끝**: 동적 내용이 변경되지 않는 한, 전체 시스템 프롬프트(정적 + 동적)가 세션 내에서 캐시 가능하다.
4. **대화 히스토리 끝**: 가장 최근 사용자 메시지까지의 대화가 현재 요청의 continuation을 위해 캐시 가능하다(모델이 여러 continuation을 필요로 하는 경우, 예: tool 사용).

중단점은 재사용 가능한 접두사 길이를 최대화하도록 배치된다. 더 긴 캐시된 접두사는 재처리해야 할 내용이 적어져 지연 시간과 비용 모두를 줄인다.

### 4.2 세션 안정적 Tool 스키마

Tool 스키마(API에 전송되는 사용 가능한 tool의 JSON 설명)는 세션 시작 시 한 번 계산되고 세션 기간 동안 고정된다. 이는 캐싱에 중요하다: tool 스키마가 요청 간에 변경되면, tool 스키마 지점 이후의 캐시된 접두사가 무효화될 것이기 때문이다.

세션 안정성 제약은 세션 내에서 tool을 동적으로 추가하거나 제거할 수 없음을 의미한다(MCP tool은 예외적으로 특별히 처리됨). 이는 Claude Code의 유스케이스에서 수용 가능한 제한이다. 사용 가능한 tool은 설치에 의해 결정되며 코딩 세션 중에 변경되지 않기 때문이다.

### 4.3 Beta 헤더 래칭

Claude Code는 캐싱 기능에 접근하기 위해 beta API 헤더를 사용한다. Beta 헤더가 활성화(요청에 포함)되면, 세션의 나머지 기간 동안 비활성화되지 않는다. 이 "래칭" 동작은 일관된 캐싱 동작을 보장한다: 요청이 beta 기능으로 캐시를 설정하면, 후속 요청이 해당 기능을 계속 사용하고 설정된 캐시를 활용할 수 있다.

대안(요청별 조건에 따라 beta 헤더를 토글)은 캐시 파편화를 유발할 것이다: beta 헤더 없는 요청이 헤더 있는 요청에 의해 설정된 캐시를 놓치게 되며, 내용이 동일하더라도 마찬가지이다.

### 4.4 프롬프트 캐시 파손 감지 (728줄)

`promptCacheBreakDetection.ts`는 프롬프트 캐시 파손 -- 요청이 예상치 않게 캐시에 적중하지 못하는 상황 -- 을 감지하고 진단하기 위한 전용 모듈(728줄)이다.

감지 알고리즘은 2단계 접근 방식을 사용한다:

**1단계: 상대적 하락 감지**

시스템은 현재 요청의 캐시 적중률을 최근 요청의 이동 평균과 비교한다. 캐시 읽기 토큰이 최근 평균 대비 5% 이상 하락하면 캐시 파손으로 플래그된다.

**2단계: 절대적 임계값**

상대적 하락만으로는 정상적 변동(예: 작업 디렉토리 변경으로 인한 약간 다른 시스템 프롬프트)에 의해 트리거될 수 있어 불충분하다. 시스템은 캐시 파손을 플래그하기 위해 추가적으로 2,000 토큰 이상의 절대적 하락을 요구한다.

두 조건이 모두 충족되어야 한다: 상대적 하락 > 5% 그리고 절대적 하락 > 2,000 토큰. 이 이중 임계값 접근 방식은 진짜 캐시 파손을 잡으면서 오탐을 최소화한다.

캐시 파손이 감지되면 시스템은:

1. 진단 정보를 기록한다 (어떤 접두사 세그먼트가 변경되었는지, 추정 비용 영향)
2. 원인을 식별하려 시도한다 (시스템 프롬프트 변경, tool 스키마 변경, 대화 히스토리 수정)
3. 선택적으로 집계 모니터링을 위해 파손을 분석 시스템에 보고한다

728줄의 모듈은 진단 기능치고는 상당한 규모이며, Claude Code의 비용 및 지연 시간 특성에 대한 캐싱 성능의 중요성을 반영한다.

### 4.5 사용자 간 전역 캐시

시스템 프롬프트의 정체성 부분(같은 Claude Code 버전의 모든 사용자에게 동일한 정적 지침)은 전역적으로 캐시되도록 설계되어 있다. 이는 새 Claude Code 버전으로 처음 요청을 보내는 사용자가 모든 후속 사용자를 위해 캐시를 "워밍"함을 의미한다.

Anthropic API의 프롬프트 캐시에 대한 1시간 TTL(time-to-live)을 감안하면, 최소 한 명의 사용자가 시간당 하나의 요청을 하는 한(광범위하게 사용되는 tool에서는 사실상 보장됨) 전역 캐시는 효과적이다. 경제적 영향이 상당하다: 정적 시스템 프롬프트가 약 8-10K 문자(~2-3K 토큰)이며, 이를 캐싱하면 매 요청에서 해당 토큰의 재처리를 회피한다.

### 4.6 프롬프트 캐싱에 대한 비평적 평가

**잘 만든 점:**

- **SYSTEM_PROMPT_DYNAMIC_BOUNDARY**: 정적/동적 분리는 유연성을 희생하지 않고 효과적인 캐싱을 가능하게 하는 깔끔한 아키텍처 결정이다.
- **cache_edits (서버 측 삭제)**: 캐시를 무효화하지 않고 캐시된 내용을 수정하는 능력은 진정한 혁신이다. 이는 어떤 캐싱 시스템의 표준 기능도 아니며, 그 구현은 Claude Code와 Anthropic API 간의 깊은 통합을 시사한다.
- **프롬프트 캐시 파손 감지**: 캐시 파손 감지를 위한 728줄 전용 모듈은 애플리케이션 코드에서 보기 드문 수준의 운영 정교함을 보여준다. 이는 캐싱 성능 문제에 대한 실제 경험에서 나온 종류의 기능이다.
- **정체성 프롬프트의 전역 캐시**: 사용자 간 캐싱을 위해 1시간 TTL을 활용하는 것은 대규모에서 경제적으로 의미 있다.

**아쉬운 점:**

- **cache_edits API 결합**: `cache_edits` 메커니즘은 Claude Code를 Anthropic API의 캐싱 구현에 밀접하게 결합시킨다. API의 캐싱 동작이 변경되면 `cache_edits` 경로가 명확한 오류 없이 조용히 성능 저하될 수 있다.
- **TTL 민감도**: 1시간 TTL은 저사용 기간(예: 야간)에 캐싱 효과가 떨어짐을 의미한다. 시스템에는 캐시를 사전에 "프리워밍"하는 메커니즘이 없다.
- **세션 안정적 Tool 스키마**: Tool 스키마가 세션별로 고정되는 제약은 동적 tool 디스커버리를 방지하여 MCP 통합 유연성을 제한한다. MCP tool은 세션 시작 시 등록되어야 하며, 사용자가 세션 중간에 새 MCP 서버를 연결하더라도 마찬가지이다(우회 방법은 세션을 재시작하는 것이다).

## 5. 토큰 추정과 그 한계

### 5.1 4바이트당 1토큰 근사

대략적 토큰 추정 휴리스틱(4바이트당 1토큰)은 여러 핵심 시스템 -- 컨텍스트 압축 임계값, 비용 추정, tool 결과 예산 -- 에 영향을 미치므로 구체적인 분석이 필요하다.

Claude의 토크나이저(BPE -- Byte Pair Encoding 기반)에서 실제 바이트당 토큰 비율은 내용 유형에 따라 상당히 달라진다:

| 내용 유형 | 일반적 비율 | 추정 오차 |
|-----------|------------|-----------|
| 영어 산문 | 3.5-4.0 | 약간 과다 카운트 |
| Python 코드 | 3.0-3.5 | 중간 정도 과다 카운트 |
| TypeScript 코드 | 3.5-4.0 | 대략 정확 |
| JSON/YAML | 2.5-3.5 | 상당한 과다 카운트 |
| 축소된 JS | 4.5-5.5 | 중간 정도 과소 카운트 |
| 유니코드 텍스트 | 5.0-8.0 | 상당한 과소 카운트 |
| Base64 데이터 | 4.0-4.5 | 약간 과소 카운트 |

Claude Code의 주요 유스케이스(TypeScript/JavaScript 코드 읽기 및 쓰기)에 대해 4바이트당 1토큰 비율은 대략 정확하다. 하지만 JSON이 많은 tool 결과(일반적으로 많다 -- 검색 결과, 파일 목록, git 출력은 종종 JSON과 유사함)에 대해서는 과다 카운트하여, 시스템이 필요 이상으로 공격적으로 컨텍스트를 압축하게 된다.

반대로, 사용자가 비ASCII 내용(CJK 코드베이스의 유니코드 식별자, 비라틴 스크립트의 주석)으로 작업할 때 비율은 상당히 과소 카운트하여, 압축이 트리거되기 전에 컨텍스트가 오버플로할 수 있다.

### 5.2 컨텍스트 압축 타이밍에 대한 영향

추정 오차는 컨텍스트 압축이 트리거되는 시점에 영향을 미친다:

- **과다 카운트 (JSON이 많은 세션)**: 압축이 너무 일찍 트리거된다. 사용자가 불필요한 컨텍스트 압축을 경험하며, 이는 정보 손실을 유발할 수 있다. 시스템이 과도하게 보수적이지만 안전하다.
- **과소 카운트 (유니코드가 많은 세션)**: 압축이 너무 늦게 트리거된다. API가 "컨텍스트가 너무 긺" 오류를 반환할 수 있으며, 이는 agent loop에서 반응적 압축 continue point를 트리거한다. 시스템은 복구하지만 지연 패널티가 있다.

코드베이스는 이 한계를 인정하지만 내용 유형별 비율로 해결하지는 않는다. 가능한 이유는: 내용 유형 감지의 복잡성(대표 샘플의 토큰화가 필요할 것)이 더 정확한 추정의 이점보다 크며, 정확한 수치가 각 요청 후 API에서 사용 가능하고 하이브리드 추정에 사용되기 때문이다.

## 6. 메모리와 지속성

### 6.1 세션 메모리

Claude Code는 "세션 메모리"를 유지한다. 이는 컨텍스트 압축에 걸쳐 지속되는 현재 세션의 실행 요약이다. Autocompact가 실행되면 압축된 메시지의 핵심 정보가 세션 메모리에 반영되어 살아남도록 보장한다.

세션 메모리는 다음을 보존하도록 구조화되어 있다:

- **파일 수정**: 어떤 파일이 생성, 수정, 삭제되었는지와 그 이유
- **핵심 결정**: 아키텍처 결정, 기술 선택, 그 근거
- **오류 해결**: 발생한 오류와 해결 방법
- **사용자 선호**: 세션 중 표현된 선호 (예: "탭 대신 스페이스 사용")

세션 메모리는 시스템 프롬프트의 동적 부분에 포함되어, 매 요청마다 모델이 사용할 수 있도록 보장한다.

### 6.2 세션 간 메모리

세션 메모리를 넘어, Claude Code는 `~/.claude/memory.md`(또는 플랫폼별 동등물)에 저장된 지속적 세션 간 메모리를 지원한다. 이 파일은 세션 시작 시 로드되어 동적 시스템 프롬프트에 주입된다.

세션 간 메모리는 Claude Code가 세션 간에 사용자 선호와 프로젝트 관례를 "기억"할 수 있게 한다. 예를 들어, 사용자가 지속적으로 TypeScript strict 모드를 요청하면, 이 선호가 세션 간 메모리에 저장되어 향후 세션에서 자동으로 적용될 수 있다.

세션 간 메모리는 사용자가 편집할 수 있다(일반 마크다운 파일이다). 이는 투명성과 제어를 제공한다. 사용자는 언제든지 지속적 메모리를 검사, 수정, 삭제할 수 있다.

## 7. 전체적 비평적 평가

### 7.1 아키텍처적 강점

**5계층 압축 파이프라인은 정교하고 잘 설계되어 있다.** 각 계층은 서로 다른 세분성(개별 tool 결과, 메시지 세그먼트, 대화 영역, 전체 대화)을 대상으로 하며, 캐스케이드는 갑작스러운 컨텍스트 오버플로 대신 점진적 성능 저하를 보장한다.

**Context Collapse의 읽기 시점 투영 패턴은 특히 우아하다.** 전체 히스토리를 보존하고 요청 시 투영을 계산함으로써, 변이 기반 압축의 비가역성을 피하면서 효율성을 유지한다.

**프롬프트 캐싱 전략은 아키텍처에 깊이 통합되어 있다.** SYSTEM_PROMPT_DYNAMIC_BOUNDARY 센티널, 캐시 범위, 세션 안정적 tool 스키마, 캐시 파손 감지는 캐싱 효율성을 위해 처음부터 설계된 시스템을 보여준다. 사후에 덧붙인 것이 아니다.

**캐시 무효화 없는 서버 측 삭제를 위한 `cache_edits` 메커니즘은 진정으로 혁신적이다.** 이는 클라이언트와 API 간의 긴밀한 통합에서만 나올 수 있는 종류의 최적화이며, 상당한 비용 및 지연 시간 이점을 제공한다.

### 7.2 아키텍처적 우려

**5계층 파이프라인은 취약하다.** 각 계층에는 자체적인 트리거 조건, 임계값, 다른 계층과의 상호작용이 있다. Context Collapse와 autocompact 간의 90%/93%/95% 경쟁 조건은 이러한 복잡성이 가능하게 하는 미묘한 버그의 구체적 예이다. 여섯 번째 계층을 추가하거나 기존 계층의 임계값을 수정하려면 모든 계층 간 상호작용의 신중한 분석이 필요할 것이다.

**Feature flag 증식은 조합적 테스트 과제를 만든다.** 컨텍스트 관리 동작을 제어하는 15개 이상의 GrowthBook 플래그로, 가능한 구성의 수가 실질적 테스트 한계를 초과한다. 코드베이스는 소수의 "축복된" 구성이 테스트되고, 나머지는 구성적 추론에 기반하여 작동한다고 가정하는 것일 가능성이 높다.

**유출된 소스에서 Snip 계층(계층 1)이 완전히 부재하는 것은 우려된다.** 제거되었다면 참조가 정리되었어야 한다. DCE된 feature flag 뒤에 존재한다면, 유출된 소스는 이 계층을 포함한 적이 없는 구성을 나타내며, 이 계층이 파이프라인의 나머지와 얼마나 잘 상호작용하는지에 대한 의문을 제기한다.

**4바이트당 1토큰 추정은 실용적이지만 체계적 편향을 만든다.** 서로 다른 사용자 집단에 서로 다르게 영향을 미친다. 유니코드가 많은 코드베이스로 작업하는 사용자는 ASCII만 사용하는 코드베이스로 작업하는 사용자보다 더 나쁜 컨텍스트 관리를 경험할 것이다.

### 7.3 대안과의 비교

Claude Code의 컨텍스트 관리 접근 방식은 다른 AI 코딩 tool과 다르다:

- **GitHub Copilot**: 코드 검색을 통한 RAG(검색 증강 생성)를 사용하여 긴 컨텍스트 문제 자체를 회피하지만, 대화 연속성을 희생한다.
- **Cursor**: RAG와 컨텍스트 선택 UI의 조합을 사용하여, 어떤 컨텍스트가 포함되는지에 대한 명시적 제어를 사용자에게 제공한다.
- **Claude Code**: 자동 압축과 함께 긴 컨텍스트 모델을 사용하여, 압축 복잡성의 비용으로 대화 연속성을 보존한다.

Claude Code의 접근 방식이 가장 야심적이고 가장 복잡하다. 작동할 때 최상의 사용자 경험(수동 컨텍스트 관리 없는 매끄러운 긴 대화)을 제공하지만, 더 단순한 접근 방식보다 더 많은 실패 모드를 가진다.

### 7.4 관찰자를 위한 제안

유사한 시스템을 구축하는 팀에게, Claude Code의 컨텍스트 관리에서 얻을 수 있는 핵심 교훈은 다음과 같다:

1. **처음부터 캐싱을 위해 설계하라.** SYSTEM_PROMPT_DYNAMIC_BOUNDARY 패턴은 구현이 단순하면서도 상당한 이점을 제공한다. 기존 프롬프트 아키텍처에 캐싱을 나중에 적용하는 것은 훨씬 더 어렵다.

2. **압축에 변이 대신 읽기 시점 투영을 사용하라.** 전체 히스토리를 복원하는 능력은 디버깅과 사용자 경험에 가치 있으며, 투영은 변이보다 안전하다.

3. **다계층 토큰 예산을 구현하라.** 컨텍스트 관리를 위한 단일 임계값은 불충분하다. 서로 다른 전략을 가진 다중 임계값(초기 단계는 경량, 후기 단계는 공격적)이 더 나은 사용자 경험을 제공한다.

4. **캐시 파손 감지에 투자하라.** 프롬프트 캐싱은 캐시 파손을 감지하고 진단할 수 있을 때만 효과적이다. 모니터링 없이는 캐싱 성능이 조용히 저하될 수 있다.

5. **추정 부정확성을 인정하고 문서화하라.** 4바이트당 1토큰 비율은 실용적인 선택이지만, 그 한계는 문서화되고 임계값 설계에 반영되어야 한다.
