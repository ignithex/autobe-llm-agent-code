# Claude Code 심층 분석: Coordinator 모드, 멀티 에이전트 패턴, 전체 아키텍처

> Claude Code의 상위 아키텍처, 멀티 에이전트 조율, 횡단적 엔지니어링 패턴에 대한 기술 리뷰이다. 이 문서는 시리즈의 세 번째이자 마지막 보고서로, agent loop 분석(보고서 1)과 컨텍스트 관리 분석(보고서 2)을 기반으로 한다. 모든 평가는 유출된 소스(512,000줄)에서 관찰 가능한 코드 패턴에 근거한다.

---

## 1. Coordinator 모드: 팀 리더로서의 LLM

### 1.1 아키텍처 개요

Coordinator 모드는 Claude Code의 멀티 에이전트 오케스트레이션 접근 방식이다. 고정된 에이전트 파이프라인이나 엄격한 워크플로 엔진을 사용하는 대신, Claude Code는 오케스트레이션 결정을 LLM 자체에 위임한다. 주 모델이 하위 "워커" 에이전트를 생성하고, 작업을 할당하며, 결과를 종합하는 "팀 리더" 역할을 한다.

이것은 오케스트레이션 토폴로지가 코드에 정의되는 전통적 멀티 에이전트 시스템(AutoGen, CrewAI, LangGraph 등)과 근본적으로 다른 접근 방식이다. Claude Code에서 토폴로지는 창발적이다: LLM이 대화 컨텍스트와 사용자 요청의 복잡성에 기반하여 얼마나 많은 워커를 생성할지, 어떤 작업을 할당할지, 언제 결과를 종합할지를 결정한다.

아키텍처 철학은 다음과 같이 요약할 수 있다: **LLM은 추론 엔진일 뿐 아니라 오케스트레이션 엔진이기도 하다.**

### 1.2 워커 통신: XML Task-Notification 형식

워커는 대화에 내장된 구조화된 XML 형식을 통해 coordinator와 통신한다:

```xml
<task-notification>
  <worker-id>worker-3</worker-id>
  <status>completed</status>
  <summary>
    Refactored the authentication module to use JWT tokens.
    Modified files: src/auth/jwt.ts, src/auth/middleware.ts
    All tests pass.
  </summary>
  <artifacts>
    <file path="src/auth/jwt.ts" action="modified" />
    <file path="src/auth/middleware.ts" action="modified" />
  </artifacts>
</task-notification>
```

XML 형식은 JSON보다 의도적으로 선택된 것이다. 그 이유는 다음과 같다:

1. **LLM 친숙도**: 언어 모델은 특히 혼합 콘텐츠(텍스트 + 구조화된 데이터)를 가진 중첩 구조에서 올바른 형식의 XML을 JSON보다 더 안정적으로 생성한다.
2. **스트리밍 호환성**: XML은 점진적으로(SAX 스타일) 파싱할 수 있으며, 이는 Claude Code의 스트리밍 아키텍처와 부합한다. JSON은 파싱 전에 완전한 객체가 필요하다.
3. **사람의 가독성**: 디버그 로그에서 XML은 내장된 산문 내용을 가진 구조화된 메시지에 대해 JSON보다 가독성이 더 좋다고 볼 수 있다.

### 1.3 자기완결적 프롬프트 원칙

Coordinator 모드의 핵심 설계 원칙은 워커 프롬프트가 **자기완결적**이라는 것이다. 워커는 사용자와 coordinator 간의 전체 대화 히스토리에 접근할 수 없다. 대신 coordinator가 워커의 작업에 필요한 정보만 포함하는 집중된 프롬프트를 각 워커에게 작성한다.

이 원칙은 여러 함의를 가진다:

- **워커는 대화 참여자가 아닌 내부 신호이다**: 워커는 사용자 대면 대화에 참여하지 않는다. 워커의 출력은 coordinator가 소비하며, coordinator가 사용자 대면 응답을 종합한다. 사용자는 원시 워커 출력을 절대 보지 않는다.
- **컨텍스트 격리가 교차 오염을 방지한다**: 워커는 관련 없는 대화 컨텍스트의 영향을 받을 수 없다. 이는 워커의 출력이 관련 없는 이전 대화에 의해 채색되는 컨텍스트 혼란의 위험을 줄인다.
- **토큰 소비 감소**: 각 워커가 전체 대화보다 작은 컨텍스트로 동작하여, 비용과 컨텍스트 오버플로의 위험 모두를 줄인다.

트레이드오프는 coordinator가 워커의 프롬프트에 모든 관련 컨텍스트를 포함해야 한다는 것이다. Coordinator가 핵심 컨텍스트를 생략하면 워커는 부정확하거나 불완전한 결과를 생성할 것이다. 이는 coordinator의 프롬프트 엔지니어링에 부담을 주며, 이는 주 LLM이 처리하므로 LLM의 판단 품질에 좌우된다.

### 1.4 워커 수명 주기: 계속 vs. 새로 생성

워커가 작업을 완료하면 coordinator는 결정에 직면한다: 기존 워커에 추가 지시를 계속할 것인가, 새 워커를 생성할 것인가?

결정 휴리스틱은 다음과 같다:

- **계속**: 후속 작업이 원래 작업과 동일한 컨텍스트를 요구할 때(예: "방금 작성한 코드에 대한 테스트 추가"). 계속하면 워커의 축적된 컨텍스트를 재사용하여, 재설정 비용을 회피한다.
- **새로 생성**: 후속 작업이 원래 작업과 독립적일 때(예: "문서도 업데이트하기"). 새 워커는 새 작업에 맞춘 신선한 컨텍스트로 시작한다.

실제로 coordinator는 새 워커 생성을 선호하는 경향이 있다. 이것이 "더 안전"하기 때문이다. 새 워커는 이전 작업의 오래된 컨텍스트에 혼란될 수 없다. 생성 편향은 상태를 가진 워커 대화를 관리하는 어려움에 대한 실용적 대응이다.

### 1.5 워커에 대한 리소스 제한

워커는 폭주 실행을 방지하는 리소스 제한의 적용을 받는다:

- **토큰 예산**: 각 워커에 최대 토큰 예산(입력 및 출력 모두)이 있다. 워커가 예산을 초과하면 실행이 종료되고 coordinator가 "예산 초과" 알림을 받는다.
- **Tool 호출 제한**: 워커에 작업당 최대 tool 호출 횟수가 있다. 이는 워커가 tool 실행의 무한 루프에 진입하는 것을 방지한다.
- **시간 제한**: 워커에 최대 실행 시간이 있다. 오래 실행되는 워커는 타임아웃 알림과 함께 종료된다.
- **파일 접근 범위**: 워커는 프로젝트 디렉토리와 명시적으로 허용된 경로 내의 파일에만 접근할 수 있다. 이는 손상된 워커가 프로젝트 외부의 민감한 파일에 접근하는 것을 방지한다.

이러한 제한은 기본적으로 보수적이며, coordinator가 작업 복잡성에 따라 조정할 수 있다. Coordinator는 복잡할 것으로 예상되는 작업(예: "이것은 대규모 리팩토링이니 tool 호출 50회까지 허용")에 대해 더 높은 제한을 명시적으로 부여할 수 있다.

## 2. 멀티 에이전트 아키텍처

### 2.1 캐시 공유 Fork를 활용한 Fork Subagent 패턴

Coordinator 모드(LLM을 오케스트레이터로 사용) 외에도, Claude Code는 하위 수준의 멀티 에이전트 패턴인 **fork subagent**를 지원한다. 이 패턴은 현재 에이전트 상태(대화 히스토리, 시스템 프롬프트, tool 구성)의 복사본을 생성하고 독립적으로 실행한다. Fork는 부모의 프롬프트 캐시를 공유하지만 자체적인 대화 상태를 가진다.

캐시 공유 메커니즘은 `CacheSafeParams`를 통해 구현되며, 이는 에이전트 상태의 어떤 부분이 fork 간에 안전하게 공유될 수 있는지를 식별하는 매개변수 구조이다:

```typescript
interface CacheSafeParams {
  systemPromptHash: string;      // 캐시 가능한 시스템 프롬프트 접두사 식별
  toolSchemaHash: string;        // 캐시 가능한 tool 스키마 블록 식별
  conversationPrefix: Message[]; // 공유 대화 접두사 (불변)
}
```

Fork가 생성되면:

1. 부모의 `CacheSafeParams`가 fork에 복제된다
2. Fork의 대화가 부모와 동일한 접두사로 시작한다
3. Fork는 fork 전용 메시지를 추가하여 부모에서 분기한다
4. Fork의 API 요청은 부모와 동일한 캐시 중단점을 포함하여, 공유 접두사에 대한 캐시 적중을 가능하게 한다

이 패턴은 투기적 실행에 사용된다: 시스템이 문제에 대한 서로 다른 접근 방식을 탐색하기 위해 여러 에이전트를 fork하고, 실패한 fork를 폐기하며, 승리한 fork로 계속할 수 있다. 캐시 공유는 forking 비용을 최소화한다 -- 분기하는 접미사만 처리하면 된다.

### 2.2 Sidechain 파일을 통한 에이전트 격리

여러 에이전트(coordinator + 워커, 또는 fork된 subagent)가 동시에 작동할 때, 서로의 부작용으로부터 격리되어야 한다. 주요 관심사는 파일시스템 수정이다: 두 에이전트가 같은 파일을 동시에 수정하면 결과가 예측 불가능해진다.

Claude Code는 **sidechain 파일**로 이를 해결한다: 각 에이전트가 별도의 파일 복사본에서 작동하며, 에이전트 완료 시 결과가 병합된다. Sidechain 메커니즘:

1. **에이전트 시작 시**: 에이전트의 작업 디렉토리에 copy-on-write 파일시스템 계층이 오버레이된다. 읽기는 원본 파일로, 쓰기는 sidechain으로 간다.
2. **실행 중**: 모든 파일 수정이 원본 파일이 아닌 sidechain에 캡처된다.
3. **에이전트 완료 시**: Sidechain의 수정이 coordinator(또는 사용자)에게 리뷰 및 병합을 위해 제시된다.

Copy-on-write 오버레이는 완전한 파일시스템 가상화(비용이 많이 들고 플랫폼 의존적일 것)가 아니다. 대신 tool 수준에서 구현된다: Write와 Edit tool이 에이전트가 sidechain에서 실행 중인지 확인하고 쓰기를 적절히 리디렉션한다. 이것이 더 단순하고 이식성이 좋지만 덜 투명하다 -- 파일에 직접 쓰는 bash 명령은 sidechain을 우회할 것이다.

### 2.3 ContentReplacementState 복제

에이전트를 fork할 때, 부모의 `ContentReplacementState`(tool 결과 잘라내기를 추적함 -- 보고서 2, 1.1절 참조)가 fork에 복제되어야 한다. 이는 fork의 토큰 예산 회계가 부모의 것과 일관되도록 보장한다.

복제는 심층적이다: 각 교체 항목이 독립적으로 복사되며, 확정된 잘라내기의 "동결" 상태를 보존한다. 이는 fork가 부모가 동결한 잘라내기를 해제하여, fork의 토큰 예산이 부모의 캐시 공유 가정에서 분기하는 시나리오를 방지한다.

### 2.4 에이전트 간 동시성 제어

동시 에이전트 실행은 신중한 동시성 제어를 요구한다. Claude Code는 여러 메커니즘을 사용한다:

- **API 호출에 대한 세마포어**: 세마포어가 모든 에이전트에 걸친 동시 API 호출 수를 제한한다. 이는 병렬 요청으로 API를 압도하는 것을 방지하고 속도 제한을 준수한다.
- **파일 수준 잠금**: 에이전트가 파일에 쓸 때(직접 또는 sidechain을 통해), 파일 수준 잠금이 다른 에이전트가 같은 파일에 쓰는 것을 방지한다. 읽기는 잠기지 않는다(copy-on-write가 일관성을 보장함).
- **에이전트 내 순차적 tool 실행**: 단일 에이전트 내에서 `isConcurrencySafe`가 아닌 tool은 순차적으로 실행된다. 이는 단일 에이전트가 동시 쓰기를 통해 자체 상태를 손상시키는 것을 방지한다.

동시성 모델은 보수적이다: 처리량보다 정확성을 우선시한다. 데이터 무결성(정확한 파일 수정)이 속도보다 중요한 개발자 tool에 적절하다.

## 3. 전체 아키텍처

### 3.1 진입점과 초기화 흐름

Claude Code의 초기화는 단계적 프로세스를 따른다:

1. **CLI 파싱**: 명령줄 인자가 파싱되고 검증된다. 구성 파일이 로드된다(사용자 설정, 프로젝트 설정, 기업 설정).
2. **환경 감지**: 시스템이 운영 체제, 셸, 사용 가능한 도구(git, node, python 등), 작업 디렉토리를 감지한다.
3. **Tool 등록**: 사용 가능한 tool이 인스턴스화되고 등록된다. Tool 스키마가 계산되고 고정된다.
4. **시스템 프롬프트 조립**: 시스템 프롬프트가 정적 및 동적 섹션에서 조립되며, 섹션 경계에 캐시 중단점이 배치된다.
5. **세션 초기화**: 새 세션이 생성(또는 기존 세션이 재개)된다. 세션 메모리가 로드된다.
6. **MCP 연결**: MCP 서버가 구성되어 있으면 연결이 설정되고 MCP 제공 tool이 등록된다.
7. **Agent loop 시작**: 메인 agent loop(보고서 1)가 시작되어 사용자 입력을 처리할 준비가 된다.

초기화는 단계 수에도 불구하고 빠르게(일반적으로 2초 이내) 설계되어 있다. 대부분의 단계가 I/O 바운드(파일 읽기, 네트워크 연결)이며 가능한 곳에서 병렬화된다.

### 3.2 모듈 구성

소스 코드는 여러 최상위 디렉토리로 구성되어 있다:

```
src/
  tools/          - Tool 구현체 (Read, Write, Edit, Bash, Glob, Grep 등)
  services/       - 서비스 계층 (API 클라이언트, 캐싱, 세션 관리)
  utils/          - 유틸리티 함수 (토큰 추정, 경로 검증 등)
  constants/      - 상수 및 구성 기본값
  hooks/          - 확장성을 위한 Hook 시스템 (stop hook, 권한 hook)
  entrypoints/    - 다양한 실행 모드의 진입점 (CLI, IDE, API)
  coordinator/    - Coordinator 모드 구현
  security/       - 보안 모듈 (bash 검증기, ML 분류기, 샌드박싱)
  context/        - 컨텍스트 관리 (압축 파이프라인, 시스템 프롬프트 조립)
```

구성은 기능적 분해를 따른다: 각 디렉토리가 특정 역량에 관련된 코드를 포함한다. 횡단적 관심사(오류 처리와 로깅 등)는 상속이나 믹스인이 아닌 공유 유틸리티를 통해 처리된다.

### 3.3 구성 캐스케이드

Claude Code의 구성 시스템은 우선순위 정렬된 캐스케이드를 사용한다:

```
CLI 플래그 (최고 우선순위)
  └── 프로젝트 설정 (.claude/settings.json)
       └── 사용자 설정 (~/.claude/settings.json)
            └── 기업 설정 (MDM으로 배포)
                 └── 기본값 (최저 우선순위)
```

높은 우선순위 소스가 같은 구성 키에 대해 낮은 우선순위 소스를 오버라이드한다. 특정 키(보안 관련)는 사용자가 보안 정책을 약화시키는 것을 방지하기 위해 기업 수준에서만 설정할 수 있다.

캐스케이드는 지연 평가된다: 구성 값이 시작 시 병합되는 것이 아니라 요청 시 조회된다. 이는 에이전트를 재시작하지 않고도 동적 구성 변경(예: 세션 중간에 프로젝트 설정 변경)을 가능하게 한다.

### 3.4 MCP 통합과 플러그인 아키텍처

Model Context Protocol(MCP)은 Claude Code의 주요 확장성 메커니즘이다. MCP 서버는 Claude Code의 내장 역량을 증강하는 추가 tool과 리소스를 제공한다.

MCP 통합 세부사항:

- **Tool 디스커버리**: MCP 서버는 MCP 프로토콜을 통해 사용 가능한 tool을 광고한다. 이러한 tool은 내장 tool과 함께 등록되며 시스템 프롬프트의 tool 스키마 블록에 나타난다.
- **권한 모델**: MCP tool은 기본적으로 명시적 사용자 승인을 요구한다(MCP 서버가 명시적으로 읽기 전용으로 선언하지 않는 한 `isReadOnly`가 아님). 이는 신뢰할 수 없는 MCP 서버가 동의 없이 부작용을 실행하는 것을 방지하는 실패 시 닫힘 기본값이다.
- **캐시되지 않는 프롬프트 섹션**: MCP 서버 지침은 요청 간에 변경될 수 있으므로(세션 중간에 새 MCP 서버가 연결되는 경우) `DANGEROUS_uncachedSystemPromptSection`으로 등록된다. 이는 캐싱 비용(보고서 2, 2.3절 참조)이 있지만 정확성을 위해 필요하다.
- **리소스 프로토콜**: MCP 서버는 컨텍스트에 포함되는 리소스(파일, 데이터)도 제공할 수 있다. 리소스는 tool 결과와 동일한 토큰 예산의 적용을 받는다.

### 3.5 IDE 브릿지 (VS Code 확장 통신)

Claude Code는 독립형 CLI tool로 실행되거나 VS Code 확장의 백엔드로 실행될 수 있다. IDE 백엔드로 실행될 때 로컬 IPC 채널(Unix 도메인 소켓 또는 Windows의 명명된 파이프)을 통해 VS Code 확장과 통신한다.

IDE 브릿지는 다음을 가능하게 한다:

- **파일 동기화**: 확장이 IDE에서 파일이 열리거나 닫히거나 수정될 때 Claude Code에 알린다. Claude Code는 이 정보를 사용하여 컨텍스트 우선순위를 정할 수 있다(최근 열린 파일이 더 관련성이 높을 가능성).
- **진단 공유**: 확장이 IDE 진단(TypeScript 오류, 린터 경고)을 Claude Code와 공유하여, 코드 수정에 대한 추가 컨텍스트를 제공한다.
- **선택 컨텍스트**: 확장이 사용자의 현재 선택, 커서 위치, 보이는 범위를 공유하여, Claude Code가 관련 코드 영역에 집중할 수 있게 한다.
- **Stop hook**: 확장이 agent loop가 조기 종료하는 것을 방지하는 stop hook을 등록할 수 있다(예: 사용자가 IDE를 통해 추가 지시를 대기열에 넣은 경우).

IDE 브릿지는 선택적 향상으로 설계되어 있다: Claude Code는 브릿지 없이도 완전히 기능하며, 브릿지는 응답 품질을 개선하는 맥락적 정보만 추가한다.

## 4. 오류 처리 철학

### 4.1 "보류하고 복구하기(Withhold and Recover)" 패턴

보고서 1에서 논의한 것처럼, Claude Code의 오류 처리는 "보류하고 복구하기" 원칙을 따른다: 오류가 발생하면 시스템이 사용자에게 오류를 표시하지 않고 자동 복구를 시도한다. 복구 불가능한 오류만 사용자에게 표시된다.

이 철학은 코드베이스 전반에 걸쳐 일관되게 구현된다:

- **API 오류**: 일시적 API 오류(429, 5xx)가 지터를 포함한 지수 백오프를 트리거한다. 사용자는 짧은 "재시도 중..." 메시지를 보지만 오류 자체는 보지 않는다.
- **Tool 오류**: Tool 실행 실패가 LLM에 tool 결과로 보고되어, 모델이 접근 방식을 조정할 수 있게 한다. 사용자는 원시 오류가 아닌 모델의 조정된 접근 방식을 본다.
- **컨텍스트 오류**: 컨텍스트 오버플로가 오류 메시지가 아닌 압축(보고서 2)을 트리거한다. 사용자는 오래된 컨텍스트가 요약된 것을 알아차릴 수 있지만 명시적 오류를 보지는 않는다.
- **보안 오류**: 보안 검증기에 의해 차단된 명령이 모델에 권한 거부로 보고된다. 모델은 대안적 접근 방식을 제안하거나 사용자 승인을 요청할 수 있다.

### 4.2 지능적 오류 분류를 통한 재시도 전략

모든 오류가 재시도 가능한 것은 아니다. Claude Code는 오류를 재시도 전략을 결정하는 카테고리로 분류한다:

| 오류 카테고리 | 재시도 전략 | 예시 |
|--------------|------------|------|
| 일시적 | 지수 백오프 (최대 5회 재시도) | HTTP 429, 503, 네트워크 타임아웃 |
| 복구 가능 | 수정 후 재시도 | 컨텍스트가 너무 긺 (압축 후 재시도), max_tokens (상승 후 재시도) |
| 영구적 | 재시도 없음, 사용자에게 표시 | 인증 실패, 계정 정지 |
| 안전하지 않음 | 재시도 없음, 실행 차단 | 보안 위반, 샌드박스 탈출 시도 |

분류는 discriminated union(Claude Code의 타입 시스템 전반에 걸쳐 광범위하게 사용되는 패턴)으로 구현된다:

```typescript
type ErrorClassification =
  | { type: 'transient'; retryAfterMs: number }
  | { type: 'recoverable'; recovery: RecoveryStrategy }
  | { type: 'permanent'; userMessage: string }
  | { type: 'unsafe'; securityEvent: SecurityEvent };
```

이 분류는 재시도 불가능한 오류를 재시도하거나(시간 낭비) 재시도 가능한 오류를 사용자에게 표시하는(경험 저하) 일반적인 안티패턴을 방지한다.

### 4.3 서킷 브레이커 패턴

여러 하위 시스템이 서킷 브레이커를 구현한다. 이는 임계 횟수의 실패 후 실패하는 컴포넌트를 비활성화하여 연쇄 실패를 방지하는 메커니즘이다.

**Autocompact 서킷 브레이커**: 3회 연속 autocompact 실패(LLM이 유용한 요약을 생성하지 못함) 후, autocompact가 세션 동안 비활성화된다. 시스템은 결정적이고 같은 방식으로 "실패"할 수 없는 Context Collapse로 폴백한다.

**yoloClassifier 서킷 브레이커**: 구성 가능한 횟수의 분류기 실패(API 오류, 타임아웃) 후, ML 기반 안전 분류기가 비활성화되고 모든 모호한 명령이 수동 사용자 승인을 요구한다. 이는 분류기 장애가 모든 명령을 차단(가용성 실패)하거나 모든 명령을 조용히 승인(보안 실패)하는 것을 방지한다.

**API 서킷 브레이커**: 구성 가능한 횟수의 연속 API 실패(속도 제한이 아닌 실제 오류) 후, agent loop가 계속 재시도하는 대신 진단 메시지와 함께 종료된다. 이는 시스템이 고장난 엔드포인트에 API 할당량을 소진하는 것을 방지한다.

### 4.4 우아한 성능 저하 사례

Claude Code는 여러 수준에서 우아한 성능 저하를 보인다:

- **Tree-sitter 사용 불가**: 정규식 기반 bash 파싱으로 폴백한다. 보안이 약화되지만 기능적이다.
- **샌드박싱 사용 불가**: tool 수준 권한 검사로 폴백한다. 격리가 줄어들지만 제거되지는 않는다.
- **프롬프트 캐시 미스**: 전체 프롬프트 처리로 폴백한다. 지연 시간이 증가하지만 기능은 보존된다.
- **MCP 서버 연결 끊김**: 내장 tool만으로 폴백한다. 역량이 줄어들지만 핵심 기능은 보존된다.
- **세션 메모리 손상**: 신선한 세션으로 폴백한다. 히스토리가 유실되지만 시스템은 기능적이다.

우아한 성능 저하가 항상 명시적으로 설계된 것은 아니다. 일부는 계층적 아키텍처에서 창발적이다. 각 보안 계층이 독립적이므로, 어느 한 계층의 실패가 나머지 계층에 의해 보상된다.

## 5. 코드 품질 관찰

### 5.1 QueryDeps 주입 패턴

Agent loop 함수 `query()`는 네 가지 의존성을 캡슐화하는 `QueryDeps` 매개변수를 받는다:

```typescript
interface QueryDeps {
  apiClient: ApiClient;
  toolRegistry: ToolRegistry;
  permissionService: PermissionService;
  contextManager: ContextManager;
}
```

이 인터페이스의 의도적 좁음(의존성 4개만)은 긍정적인 설계 신호이다. Agent loop가 명확하고 한정된 외부 의존성 집합을 가지므로 테스트 가능하고(의존성을 모킹할 수 있음) 유지보수 가능하다(코드베이스의 나머지와의 상호작용 표면이 명시적이고 제한적임).

대안 -- 전역 컨텍스트나 서비스 로케이터에 접근 -- 은 agent loop의 의존성을 암묵적이고 추적 불가능하게 만들 것이다. 명시적 주입 패턴은 agent loop가 기능하기 위해 무엇이 필요한지를 즉시 명확하게 만든다.

### 5.2 `using` 키워드 (TC39 명시적 리소스 관리)

코드베이스는 리소스 관리를 위해 TC39 `using` 키워드를 사용한다:

```typescript
using handle = await acquireResource();
// handle은 블록이 종료될 때 자동으로 dispose됨
```

이것은 비교적 최근의 JavaScript/TypeScript 기능(개발 시점에 Stage 3, 현재 Stage 4)으로, C#의 `using`이나 Python의 `with`와 유사한 결정적 리소스 정리를 제공한다. `using`의 채택은 언어 진화를 적극적으로 추적하고 명확한 이점이 있을 때 새 기능을 기꺼이 채택하는 팀을 시사한다.

`using` 키워드는 Claude Code의 유스케이스에 특히 적절하다. 파일 핸들, API 연결, 샌드박스 프로세스 같은 리소스가 오류 경로에서도 안정적으로 정리되어야 하기 때문이다. `using` 없이는 정리에 `try/finally` 블록이 필요한데, 이는 장황하고 오류가 발생하기 쉽다(개발자가 추가하는 것을 종종 잊음).

### 5.3 스트리밍 폴백 정리를 위한 Tombstone 메시지

스트리밍 폴백 continue point가 활성화될 때(보고서 1, 1.3절, 포인트 5), 실패한 스트리밍 응답의 부분적 콘텐츠가 대화 히스토리에서 정리되어야 한다. Claude Code는 이를 위해 "tombstone 메시지"를 사용한다: 부분적 콘텐츠가 "이 내용은 실패한 스트림에서 온 것이며 무시되어야 한다"를 나타내는 특별한 마커 메시지로 대체된다.

Tombstone 패턴은 삭제보다 선호된다. 부분적 콘텐츠를 단순히 삭제하면 남은 메시지의 인덱스가 이동하여, 참조(예: 모델의 텍스트에서 "위의 내 응답 참조")를 무효화할 수 있기 때문이다. Tombstone은 콘텐츠를 명확히 무효로 표시하면서 인덱스를 보존한다.

### 5.4 기발한 "Wizard" 주석

코드베이스 곳곳에 "wizard" -- 특정 하위 시스템을 담당하는 내부 팀 구성원이나 페르소나 -- 를 참조하는 주석이 흩어져 있다:

```
// The context wizard says: never mutate the message history directly
// Prompt wizard approved this change (see PR #XXXX)
// Security wizard: this validator MUST run before the tree-sitter check
```

이 주석은 이중 목적을 수행한다: 설계 결정을 특정 전문 영역에 귀속시키고(코드 수정 시 누구에게 상의해야 하는지를 명확하게 함), 그렇지 않으면 건조한 기술 문서에 약간의 개성을 더한다. "wizard" 관례는 개별 파일이 아닌 개별 코드 결정 수준에서 작동하는 코드 소유권 주석(GitHub의 CODEOWNERS 파일 같은)의 가벼운 버전이다.

### 5.5 Bun:Bundle DCE를 활용한 Feature Flag 아키텍처

Claude Code는 GrowthBook feature flag를 광범위하게 사용하며, 빌드 시스템은 비활성화된 기능 경로의 사용하지 않는 코드 제거(DCE)를 위해 Bun의 번들러를 활용한다.

메커니즘:

1. Feature flag가 대상 배포 구성에 기반하여 번들 시점에 평가된다
2. 비활성화된 기능 경로가 Bun의 트리 셰이킹에 의해 도달 불가능한 코드로 인식된다
3. 번들러가 프로덕션 번들에서 도달 불가능한 코드를 제거한다

이것은 특정 기능(압축 파이프라인의 Snip 계층 같은)이 유출된 소스에서 부재한 이유를 설명한다: 유출된 번들을 빌드하는 데 사용된 구성에서 비활성화되었고, DCE가 구현을 완전히 제거한 것이다.

DCE 접근 방식에는 장단점이 있다:

**장점**:
- 더 작은 번들 크기 (사용하지 않는 코드 없음)
- 더 빠른 시작 (파싱하고 로드할 미사용 코드 없음)
- 보안 (비활성화된 기능이 실수로 호출될 수 없음)

**단점**:
- 유출된 소스가 실제 코드베이스의 불완전한 표현이다. 소스 리포지토리에 존재하지만 이 번들 구성에서 비활성화된 기능은 유출된 코드에서 보이지 않는다.
- 프로덕션 문제 디버깅은 특정 번들에서 어떤 기능이 활성화되었는지를 알아야 한다. 개발 환경(모든 기능 활성화)에서 작동하는 기능이 프로덕션(일부 기능이 DCE됨)에서 실패할 수 있다.

## 6. 횡단적 아키텍처 패턴

### 6.1 도메인 언어로서의 Discriminated Union

Claude Code는 도메인 모델링 tool로서 TypeScript discriminated union을 광범위하게 사용한다:

```typescript
type ToolResult =
  | { type: 'success'; content: string }
  | { type: 'error'; error: ToolError }
  | { type: 'permission_denied'; tool: string; reason: string }
  | { type: 'budget_exceeded'; limit: number; actual: number };
```

이 패턴은 여러 이점을 제공한다:

- **완전성 검사**: TypeScript의 내로잉이 모든 변형이 처리되도록 보장한다. 유니온에 새 변형을 추가하면 처리되지 않은 모든 switch/if 지점에서 컴파일 타임 오류가 발생한다.
- **자기 문서화**: 유니온 타입이 모든 가능한 결과의 간결한 사양이다. 케이스를 열거하기 위한 별도 문서가 필요 없다.
- **패턴 매칭**: 판별자에 대한 TypeScript의 `switch`가 인체공학적 패턴 매칭을 제공하여 코드를 읽기 쉽고 유지보수하기 쉽게 만든다.

이 패턴은 오류 분류, 이벤트 타입, tool 결과, 보안 결정, 그리고 코드베이스 전반의 많은 다른 도메인에 나타난다. 일관된 사용은 코드를 예측 가능하게 만드는 통합된 모델링 언어를 생성한다: 새 도메인 개념을 만나면 discriminated union으로 모델링되어 있으리라 기대할 수 있다.

### 6.2 이벤트 기반 상태 전파

Agent loop와 tool 시스템은 이벤트를 사용하여 상태 변경을 소비자(CLI 출력, IDE 확장, 분석)에 전파한다:

```typescript
yield { type: 'tool_start', tool: 'Read', input: { path: '/foo/bar.ts' } };
// ... tool 실행 ...
yield { type: 'tool_end', tool: 'Read', result: { content: '...' }, durationMs: 42 };
```

이벤트 기반 접근 방식은 agent loop를 소비자로부터 분리한다. 루프는 이벤트가 터미널에 표시되는지, IDE에 전송되는지, 분석 서비스에 기록되는지 알지 못한다. 이 분리는 다음을 가능하게 한다:

- **다중 소비자**: 같은 이벤트 스트림이 CLI, IDE, 분석에 의해 동시에 소비될 수 있다.
- **소비자 유연성**: 각 소비자가 이벤트를 독립적으로 필터, 변환, 집계할 수 있다.
- **테스트 가능성**: 이벤트 스트림이 테스트에서 캡처되고 검증될 수 있어, agent loop의 동작에 대한 완전한 기록을 제공한다.

### 6.3 중요한 곳에서의 불변성

코드베이스가 전체적으로 불변은 아니지만(JavaScript/TypeScript는 기본적으로 불변성을 강제하지 않음), 핵심 데이터 구조에 불변성 규율을 적용한다:

- **시스템 프롬프트**: 조립 후 고정되어 세션 중 우발적 변이를 방지한다.
- **Tool 스키마**: 계산 후 고정되어 캐시 안정성을 보장한다.
- **ContentReplacementState 동결 항목**: 잘라내기가 확정되면 해제할 수 없다.
- **CacheSafeParams**: 에이전트를 fork할 때 참조가 아닌 복제하여, 에이전트 간 변이를 방지한다.

불변성의 선택적 적용은 실용적 접근 방식을 반영한다: 변이가 미묘한 버그를 유발할 곳(캐싱, 에이전트 간 상태)에서는 불변성을 강제하지만, 안전하고 편리한 곳(로컬 변수, 임시 데이터 구조)에서는 변이를 허용한다.

## 7. 아키텍처적 약점과 기술 부채

### 7.1 진화적 복잡성

가장 중요한 아키텍처적 우려는 진화적 복잡성이다. 각각 특정 문제를 해결하기 위해 추가된 기능, 플래그, 복구 경로의 축적이 전체적으로 추론하기 어려운 시스템을 만든다.

구체적 양상:

- **Agent loop의 7개 continue point**: 각각 특정 엣지 케이스를 처리하기 위해 추가되었지만, 수십 개의 가능한 전이를 가진 암묵적 상태 머신을 생성한다(보고서 1, 1.7절).
- **5계층 컨텍스트 압축**: 각 계층이 서로 다른 세분성을 대상으로 하지만, 상호작용(90%/93%/95% 경쟁 조건)이 미묘하고 문서화가 부족하다(보고서 2, 1.6절).
- **23개 bash 검증기**: 각각 특정 공격 벡터를 대상으로 하지만, 순서 의존성과 지연된 결과 상호작용이 취약하다(보고서 1, 3.1절).
- **8개 소스 권한 시스템**: 각 소스가 다른 이해관계자를 다루지만, 우선순위 평가가 복잡하고 사용자가 예측하기 어렵다(보고서 1, 3.2절).

### 7.2 Agent Loop에서의 가변 상태 누출

선택적 불변성 규율(6.3절)에도 불구하고, agent loop(`query.ts`)에는 반복 간에 지속되는 여러 가변 상태 변수가 있다:

- 메시지 히스토리 배열 (tool 결과 추가와 컨텍스트 압축에 의해 변이됨)
- 토큰 예산 카운터 (추정과 API 응답 처리에 의해 변이됨)
- Continue point 트리거 (루프의 각 phase에 의해 변이됨)
- Feature flag 평가 (캐시되어 잠재적으로 오래될 수 있음)

이러한 가변 변수는 루프 반복 간 암묵적 결합을 생성한다. 한 반복의 버그가 후속 반복에 영향을 미치는 상태를 손상시킬 수 있으며, 손상은 여러 반복 후에야 나타날 수 있다. 이는 오래 지속되는 가변 상태의 전형적 증상이며, agent loop 문제의 디버깅을 특히 어렵게 만든다.

### 7.3 프로덕션 경로의 _DEPRECATED 함수

코드베이스에는 `_DEPRECATED` 접미사로 표시되었지만 프로덕션 코드 경로에서 여전히 호출되는 함수가 포함되어 있다. 사용 중단된 토큰 추정 메서드, 레거시 tool 결과 포맷팅 함수, 대체되었지만 제거되지 않은 오래된 보안 검증기 등이 그 예이다.

사용 중단되었지만 활성인 코드의 위험은 두 가지이다:

1. **유지보수 부담**: 사용 중단된 코드도 팀이 제거하려는 의도에도 불구하고 여전히 유지보수, 테스트, 보안이 필요하다. 이는 낭비된 노력을 생성한다.
2. **동작 분기**: 사용 중단된 함수와 대체 함수가 공존하면 시간이 지남에 따라 동작이 분기될 수 있다. 사용 중단된 함수를 사용하는 호출자는 대체 함수를 사용하는 호출자와 다르게 동작하여 불일치를 만든다.

가장 가능성 있는 설명은 마이그레이션 피로이다: 깊이 통합된 함수를 교체하려면 모든 호출자를 업데이트하고 새 동작을 테스트해야 하며, 이는 빠르게 진화하는 코드베이스에서 상당한 노력이 필요하다. 사용 중단 마크는 즉각적인 조치 항목이 아닌 "향후 작업" 표시자 역할을 한다.

### 7.4 Feature Flag 조합적 폭발

컨텍스트 관리와 보안 시스템만으로도 동작을 제어하는 15개 이상의 GrowthBook feature flag가 있어, 가능한 구성의 수가 천문학적이다(2^15 = 32,768). 모든 조합을 테스트하는 것은 불가능하며, 플래그 간 상호작용이 항상 문서화되어 있지도 않다.

예를 들어, `cache_edits` microcompact를 활성화하는 feature flag(보고서 2, 1.3절)가 autocompact 임계값을 변경하는 feature flag(보고서 2, 1.5절)와 어느 쪽 플래그의 설계자도 예상하지 못한 방식으로 상호작용할 수 있다. 상호작용이 무해할 수도(플래그가 독립적) 있고 병리적일 수도(두 플래그 모두 활성화하면 경쟁 조건 유발) 있다.

실제 완화 방법은 테스트되는 소수의 "축복된" 구성이 있고, 다른 조합은 플래그가 "대체로 독립적"이라는 가정 하에 안전하다고 보는 것일 가능성이 높다. 이 가정은 유지될 때까지 유지되며, 결과적 버그는 특정 플래그 조합에서만 나타나므로 진단하기 어렵다.

## 8. 다른 AI 코딩 에이전트 아키텍처와의 비교

### 8.1 Claude Code vs. 전통적 멀티 에이전트 프레임워크

| 측면 | Claude Code | AutoGen / CrewAI / LangGraph |
|------|-------------|-------------------------------|
| 오케스트레이션 | LLM 구동 (창발적) | 코드 정의 (명시적) |
| 에이전트 토폴로지 | 동적 (coordinator가 결정) | 정적 (코드에 정의) |
| 통신 | XML task-notification | 함수 호출 / 메시지 전달 |
| 상태 관리 | 격리된 공유 컨텍스트 | 에이전트별 상태, 명시적 공유 |
| 캐시 최적화 | 깊은 API 통합 | 일반적 (캐시 인식 없음) |
| 오류 복구 | 내장 (7개 continue point) | 애플리케이션 수준 |

Claude Code의 접근 방식은 더 유연하지만(LLM이 작업에 맞게 에이전트 토폴로지를 적응시킬 수 있음) 덜 예측 가능하다(토폴로지가 LLM의 판단에 의존함). 전통적 프레임워크는 경직성의 비용으로 더 많은 제어를 제공한다.

### 8.2 Claude Code vs. Cursor / Windsurf

| 측면 | Claude Code | Cursor / Windsurf |
|------|-------------|-------------------|
| 컨텍스트 전략 | 압축과 함께 긴 컨텍스트 | RAG + 선택적 컨텍스트 |
| 컨텍스트에 대한 사용자 제어 | 자동 (최소 사용자 입력) | 수동 (사용자가 파일 선택) |
| 다중 파일 편집 | sidechain으로 순차적 | diff 미리보기로 병렬 |
| 보안 모델 | 6개 이상 계층의 심층 방어 | 권한 기반 |
| 확장성 | MCP 프로토콜 | 플러그인 API |

Claude Code의 자동 컨텍스트 관리는 더 편리하지만 덜 투명하다. Cursor/Windsurf 사용자는 어떤 컨텍스트가 포함되는지를 정확히 볼 수 있어, 모델이 예상치 않은 결과를 생성할 때 디버깅에 도움이 된다.

### 8.3 아키텍처적 독창성

Claude Code 아키텍처의 여러 측면이 AI 코딩 tool 중 독특하다:

1. **스트리밍 tool 실행**: 다른 주요 AI 코딩 tool은 모델 스트리밍 중에 tool을 실행하지 않는다. 이는 진정한 지연 시간 혁신이다.
2. **서버 측 캐시 편집**: 무효화 없이 캐시된 컨텍스트를 수정하는 `cache_edits` 메커니즘은 참신하며 API와 깊이 통합되어 있다.
3. **Policy-as-type**: TypeScript 타입 이름을 보안 정책 선언으로 사용하는 것은 창의적이고 독특하다.
4. **ML 분류를 포함한 심층 방어**: bash 명령 안전을 위한 규칙 기반 검증기, AST 기반 분석, ML 기반 분류의 조합은 어떤 비교 가능한 tool보다 포괄적이다.
5. **컨텍스트 collapse를 위한 읽기 시점 투영**: 컨텍스트 압축에 비변이 투영을 사용하여 복구를 위해 전체 히스토리를 보존하는 것은 다른 tool이 배울 수 있는 우아한 접근 방식이다.

## 9. 종합 평가

### 9.1 엔지니어링 강점

Claude Code는 여러 모범 사례를 보여주는 기술적으로 인상적인 시스템이다:

- **원칙적 의존성 주입**: `QueryDeps` 패턴이 agent loop를 테스트 가능하게 하고 의존성을 명시적으로 만든다.
- **포괄적 오류 분류**: 오류 분류에 대한 discriminated union 접근 방식이 과도한 재시도와 과소 표시의 일반적 안티패턴을 방지한다.
- **탁월한 보안 문서 문화**: "wizard" 주석, policy-as-type 패턴, 보안 결정에 대한 인라인 설명이 코드베이스를 그 규모에 비해 비정상적으로 감사 가능하게 만든다.
- **깊은 캐싱 통합**: 다중 범위 캐싱 전략, 캐시 파손 감지, `cache_edits` 메커니즘은 캐싱 효율성을 위해 처음부터 설계된 시스템을 보여준다.
- **실패 시 닫힘 기본값**: tool 시스템과 보안 아키텍처 전반에 걸친 제한적 기본값의 일관된 적용이 전체 범주의 버그를 제거한다.

### 9.2 엔지니어링 우려

- **진화적 복잡성이 한계에 접근하고 있다.** 1,730줄의 agent loop, 2,450줄의 bash 보안 모듈, 5계층 압축 파이프라인은 각각 개별적으로 관리 가능하지만 전체적으로 상호작용 표면이 추론하기 어려운 시스템을 생성한다. 주기적인 아키텍처 단순화 -- 암묵적 상태 머신을 명시적으로 추출하고, 변이 전략을 통합하며, feature flag를 정리 -- 가 도움이 될 것이다.
- **Agent loop의 가변 상태는 부담이다.** `query.ts`의 오래 지속되는 가변 변수가 반복 간 암묵적 결합을 생성하여 디버깅을 어렵게 만든다. 더 함수형 스타일(각 반복이 불변 상태를 받고 새 상태를 반환)로의 리팩토링이 안정성을 개선할 것이다.
- **Feature flag 증식은 테스트 격차를 만든다.** 가능한 구성의 수가 실질적 테스트 한계를 초과한다. Feature flag에 대한 더 규율적인 접근 -- 졸업한 기능의 플래그 제거, 플래그 상호작용 문서화, 축복된 구성의 명시적 테스트 -- 이 위험을 줄일 것이다.
- **프로덕션 경로의 _DEPRECATED 함수는 기술 부채이다.** 팀은 이러한 사용 중단 마크가 약속하는 마이그레이션 완료를 우선시해야 한다. 각 사용 중단되었지만 활성인 함수는 유지보수 부담이자 동작 불일치의 잠재적 원천이다.

### 9.3 최종 판단

Claude Code는 빠르지만 신중한 엔지니어링의 특징을 보여주는 512,000줄의 코드베이스이다. 아키텍처가 원칙적이고(의존성 주입, discriminated union, 실패 시 닫힘 기본값), 보안이 포괄적이며(6개 이상 계층의 심층 방어), 성능 최적화가 혁신적이다(스트리밍 tool 실행, 서버 측 캐시 편집, 다중 범위 프롬프트 캐싱).

동시에 시스템은 자체 성공의 부담을 보여준다. 각각의 새로운 역량 -- coordinator 모드, 컨텍스트 압축, 보안 검증기, 캐시 최적화 -- 이 개별적으로는 정당화되지만 전체적으로는 부담이 되는 복잡성을 더한다. 팀은 강력한 관례("wizard" 주석 문화, policy-as-type 패턴, 실패 시 닫힘 기본값)를 통해 이 복잡성을 관리해 왔지만, 관례는 아키텍처 제약보다 약하다.

Claude Code의 다음 진화는 "관례와 함께 기능 추가"에서 "패턴을 아키텍처로 추출"로의 전환이 필요할 것이다. Agent loop의 암묵적 상태 머신이 명시적이 되어야 한다. 5계층 압축 파이프라인이 형식적으로 정의된 계층 상호작용을 가져야 한다. Feature flag 시스템이 문서화된 상호작용 매트릭스를 가져야 한다.

이것은 비판이 아니다 -- 성공적이고 빠르게 진화하는 시스템의 정상적이고 예상되는 궤적이다. Claude Code의 엔지니어링 팀은 진정으로 인상적인 것을 만들었으며, 그들이 직면하는 아키텍처적 과제는 실패가 아닌 성공의 과제이다.
